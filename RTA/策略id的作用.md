- 策略id是用来做实验的。

--- 
# 核心结论

**策略 ID 在媒体侧的本质作用，是把广告主侧的“实时决策结果”结构化地传递给媒体侧，用于后续的投放控制、出价融合、流量分层、数据归因和策略评估。**  
它通常不是媒体侧生成的，而是由**广告主 RTA 策略系统**根据用户、场景、广告计划、模型分数、业务目标等条件实时匹配后生成或选择，并在 RTA 响应中返回给媒体。

---

# 1. 策略 ID 在媒体侧有什么作用？

## 1.1 表达广告主侧的投放意图

媒体侧看到的是一次广告请求，例如：

```json
{
  "user_id": "xxx",
  "slot_id": "feed_001",
  "city": "beijing",
  "device": "android"
}
```

但广告主侧经过 ID Mapping 和用户价值判断后，可能知道这个用户是：

- 高价值老客
- 加购未购买用户
- 新客高潜人群
- 沉睡召回用户
- 大促高意向用户
- 风险或低价值用户

**策略 ID 就是广告主把这些判断结果编码后告诉媒体侧的方式。**

例如：

```json
{
  "accept": true,
  "strategy_id": "JD_RTA_10023",
  "bid_coefficient": 1.45
}
```

媒体侧不一定需要知道 `JD_RTA_10023` 背后的全部业务逻辑，但可以知道：

> 这次请求命中了广告主侧某个有效策略，并且该策略对应较高出价意愿。

---

## 1.2 参与媒体侧出价融合

在 RTA 中，媒体侧通常不会只看广告主返回的一个字段，而是会综合多个因素：

```text
最终排序分 = 广告主基础出价 × 出价系数 × 媒体侧 pCTR × 质量分 × 流量调控因子
```

其中，**策略 ID 可以作为出价系数的解释和管控依据**。

例如：

| 策略 ID | 策略含义 | 出价系数 | 媒体侧理解 |
|---|---|---:|---|
| S001 | 高价值老客复购 | 1.8 | 广告主强烈希望拿量 |
| S002 | 新客拉新 | 1.2 | 正常扩量 |
| S003 | 低价值探索 | 0.6 | 低价参与 |
| S004 | 风险用户过滤 | 0 | 不参与 |

媒体侧可以根据 `strategy_id + bid_coefficient` 判断广告主对该流量的真实价值偏好。

---

## 1.3 用于流量分层和投放控制

媒体侧可以基于策略 ID 做流量分层，例如：

| 媒体侧动作 | 策略 ID 的作用 |
|---|---|
| 是否放行参与竞价 | 某些策略 ID 可直接拒投或低优先级 |
| 是否进入特定广告计划 | 不同策略 ID 映射不同 campaign / adgroup |
| 是否走特殊排序逻辑 | 高价值策略可进入高优队列 |
| 是否限制频控 | 不同策略可配置不同曝光频次 |
| 是否控制预算消耗 | 大促策略、拉新策略、召回策略可走不同预算池 |
| 是否参与实验 | 策略 ID 可作为实验分组维度 |

例如：

```text
strategy_id = S_NEW_USER_HIGH_VALUE
```

媒体侧可以将其映射到“新客高潜拉新广告计划”，并应用对应预算、频控和创意规则。

---

## 1.4 用于数据归因和效果分析

策略 ID 的另一个重要作用是**后验分析**。

没有策略 ID 时，媒体侧和广告主侧只能看到整体投放效果：

```text
曝光 1000 万
点击 20 万
转化 1 万
ROI 1.5
```

有策略 ID 后，可以拆解到策略粒度：

| 策略 ID | 曝光 | 点击率 | 转化率 | CPA | ROI |
|---|---:|---:|---:|---:|---:|
| S001 高价值老客 | 100万 | 3.5% | 8.0% | 20 | 4.2 |
| S002 新客拉新 | 300万 | 2.1% | 3.2% | 60 | 1.8 |
| S003 沉睡召回 | 200万 | 1.8% | 2.5% | 80 | 1.2 |
| S004 探索流量 | 400万 | 0.8% | 0.6% | 200 | 0.5 |

这样广告主和媒体可以判断：

- 哪类人群值得加预算
- 哪类策略应降价
- 哪类策略需要停止
- 哪类模型分数区间效果更好
- 哪类媒体流量适合某个业务目标

---

## 1.5 用于媒体与广告主之间的对账和问题排查

RTA 链路通常涉及：

```text
媒体请求 → 广告主 RTA 判断 → 返回策略 ID → 媒体竞价 → 曝光 → 点击 → 转化回传
```

策略 ID 可以作为链路追踪字段，帮助定位问题。

例如：

| 问题 | 策略 ID 的排查价值 |
|---|---|
| 曝光量突然下降 | 是某几个策略 ID 返回量下降，还是整体 RTA 超时？ |
| ROI 下降 | 是高价值策略失效，还是探索策略占比过高？ |
| 出价异常 | 是某策略 ID 的出价系数配置过高？ |
| 转化回传异常 | 是某策略 ID 的转化链路缺失？ |
| 预算消耗过快 | 是某策略 ID 命中量异常放大？ |

所以，**策略 ID 是 RTA 投放中的核心观测维度之一**。

---

# 2. 策略 ID 是如何生成的？

策略 ID 的生成通常有两种含义：

1. **策略定义阶段生成**：运营、算法或投放系统创建策略时生成一个唯一 ID。
2. **实时请求阶段选择**：RTA 请求到来时，系统判断当前用户命中哪个策略，并返回对应策略 ID。

---

# 3. 策略 ID 的生成链路

## 3.1 第一步：定义策略

广告主侧会先在策略平台或投放平台中定义策略。

一个策略通常包括：

| 组成部分 | 示例 |
|---|---|
| 策略名称 | 高价值老客复购策略 |
| 业务目标 | 提升复购、拉新、召回、促活 |
| 适用广告计划 | 京东某投放计划 / 某大促计划 |
| 人群条件 | 近 30 天购买过、近 7 天浏览未购 |
| 模型条件 | pCVR > 0.08，LTV 分位数 Top 20% |
| 场景条件 | 信息流、开屏、搜索广告位 |
| 时间条件 | 大促期、晚高峰、周末 |
| 出价规则 | 系数 1.5 或模型动态出价 |
| 频控规则 | 24 小时最多曝光 3 次 |
| 排除规则 | 已购买用户、风险用户、投诉用户 |
| 实验分组 | A 组、B 组、模型 V1、模型 V2 |

策略创建完成后，系统会生成一个唯一标识：

```text
strategy_id = S_20260623_000123
```

或者：

```text
strategy_id = 100123
```

---

## 3.2 第二步：策略 ID 唯一化

策略 ID 必须满足几个要求：

| 要求 | 说明 |
|---|---|
| **唯一性** | 不能和其他策略冲突 |
| **稳定性** | 同一策略在有效期内不能频繁变化 |
| **可追踪性** | 能关联到策略配置、模型版本、实验版本 |
| **可解析性** | 内部系统可以根据 ID 查到策略详情 |
| **可灰度** | 支持版本升级、AB 实验、流量切分 |
| **可下线** | 策略失效后可以停止返回 |

生成方式可以是：

### 方式一：数据库自增 ID

```text
100001
100002
100003
```

优点是简单，适合单中心策略平台。  
缺点是跨系统、跨机房、跨环境时需要避免冲突。

---

### 方式二：分布式 ID

例如使用类似 Snowflake 的分布式 ID：

```text
738492847239482368
```

优点是适合高并发、多机房、多业务线。  
缺点是可读性较差，通常需要后台查询解释。

---

### 方式三：业务编码型 ID

```text
RTA_NEW_HIGH_20260623_A
RTA_RECALL_7D_V2
RTA_CART_NOT_PAY_V3
```

优点是可读性强。  
缺点是编码规范复杂，长期维护成本较高。

---

### 方式四：策略平台统一生成

这是较常见的方式。

```text
策略平台创建策略
        ↓
系统生成 strategy_id
        ↓
策略配置发布
        ↓
RTA 服务加载策略
        ↓
实时请求命中后返回 strategy_id
```

在实际系统设计中，策略 ID 往往由策略平台统一管理，下发给执行模块，并在请求链路中透传，以支持会话级或请求级策略执行、个性化干预和效果分析。类似机制中，策略 ID 可由策略平台生成，并在业务接口调用时携带，从而支撑“千人千面”的策略执行。[〔1〕](https://www.bilibili.com/read/cv26227391/)

---

# 4. 实时 RTA 请求中，策略 ID 是如何被选出来的？

## 4.1 请求进入广告主 RTA 服务

媒体侧请求广告主：

```json
{
  "request_id": "req_001",
  "media_user_id": "m_abc",
  "device_id_hash": "xxx",
  "slot_id": "feed_001",
  "city": "beijing",
  "timestamp": 1782180000
}
```

---

## 4.2 广告主做 ID Mapping

广告主先判断这个媒体用户是否能映射到广告主用户：

```text
media_user_id / device_id
        ↓
advertiser_user_id
```

如果能识别用户，就可以调用用户画像、行为特征、会员状态、历史购买、兴趣偏好等信息。

---

## 4.3 特征查询与模型预估

广告主侧会查询或实时计算：

| 特征类别 | 示例 |
|---|---|
| 用户特征 | 年龄段、城市、会员等级、购买力 |
| 行为特征 | 浏览、搜索、加购、收藏、购买 |
| 商品偏好 | 家电、手机、服饰、生鲜 |
| 生命周期 | 新客、老客、沉睡、流失 |
| 实时上下文 | 时间、媒体、广告位、网络、设备 |
| 模型分数 | pCTR、pCVR、pGMV、LTV、流失概率 |

---

## 4.4 策略匹配

系统根据规则和模型结果匹配策略。

例如：

```text
如果用户近 7 天加购未购买
且 pCVR > 0.05
且广告位为信息流
则命中策略 S_CART_7D_HIGH
```

或者：

```text
如果用户是新客
且 LTV 预估 Top 10%
则命中策略 S_NEW_USER_TOP10
```

---

## 4.5 策略优先级决策

一个用户可能同时命中多个策略：

| 命中策略 | 含义 | 优先级 |
|---|---|---:|
| S_NEW_USER | 新客策略 | 80 |
| S_HIGH_LTV | 高 LTV 策略 | 90 |
| S_BIG_PROMO | 大促策略 | 95 |
| S_RISK_FILTER | 风险过滤 | 100 |

通常需要通过优先级、互斥规则、预算状态、实验分桶来决定最终返回哪个策略 ID。

常见规则：

```text
风险过滤 > 预算控制 > 大促策略 > 高价值策略 > 普通召回 > 探索策略
```

最终返回：

```json
{
  "accept": true,
  "strategy_id": "S_BIG_PROMO",
  "bid_coefficient": 1.8
}
```

---

# 5. 策略 ID 和出价系数是什么关系？

**策略 ID 说明“为什么投”，出价系数说明“愿意花多大力度投”。**

二者关系可以理解为：

| 字段 | 含义 | 示例 |
|---|---|---|
| strategy_id | 命中的投放策略 | 高价值老客复购 |
| bid_coefficient | 对应的出价强度 | 1.6 |
| accept | 是否参与投放 | true |

例如：

```json
{
  "accept": true,
  "strategy_id": "S_OLD_USER_HIGH_VALUE",
  "bid_coefficient": 1.6
}
```

含义是：

> 这个用户命中了“高价值老客复购策略”，广告主愿意按基础出价的 1.6 倍参与竞争。

---

# 6. 媒体侧拿到策略 ID 后会怎么用？

## 6.1 映射到媒体侧投放计划

媒体侧可能维护一张映射表：

| 广告主策略 ID | 媒体侧计划 | 媒体侧动作 |
|---|---|---|
| S_NEW_HIGH | 新客拉新计划 | 参与竞价 |
| S_OLD_RECALL | 老客召回计划 | 参与竞价 |
| S_RISK | 风险过滤计划 | 不投 |
| S_EXPLORE | 探索计划 | 低价投放 |

---

## 6.2 参与最终 eCPM 排序

媒体侧可能这样计算：

```text
广告主返回价 = base_bid × bid_coefficient
媒体排序分 = 广告主返回价 × pCTR × quality_score
```

策略 ID 本身不一定直接进入公式，但它会影响：

- 使用哪个 base_bid
- 使用哪个 bid_coefficient
- 进入哪个广告计划
- 是否允许竞价
- 是否进入实验桶
- 是否应用特殊频控

---

## 6.3 做预算与频控

不同策略 ID 可以对应不同预算池：

| 策略 ID | 预算池 | 频控 |
|---|---|---|
| S_NEW_USER | 拉新预算 | 每用户每日 3 次 |
| S_RECALL | 召回预算 | 每用户每日 2 次 |
| S_BIG_PROMO | 大促预算 | 每用户每日 5 次 |
| S_LOW_VALUE | 探索预算 | 每用户每日 1 次 |

---

## 6.4 做效果回传和报表分析

媒体侧可以按策略 ID 回传曝光、点击、转化数据：

```json
{
  "request_id": "req_001",
  "strategy_id": "S_NEW_HIGH",
  "event": "conversion",
  "timestamp": 1782183600
}
```

广告主侧拿到后，可以评估：

- 哪个策略 ROI 最高
- 哪个策略 CPA 偏高
- 哪个策略需要降价
- 哪个策略需要扩量
- 哪个策略人群质量下降

---

# 7. 策略 ID 设计建议

## 7.1 不建议把太多业务含义直接写进 ID

不推荐：

```text
RTA_BEIJING_ANDROID_MALE_25_35_HIGH_LTV_CART_7D_V3
```

原因：

- ID 过长
- 隐私风险更高
- 策略变更后难维护
- 媒体侧不需要知道过多细节

更推荐：

```text
S100238
```

然后广告主内部维护元数据：

| strategy_id | 策略名称 | 人群 | 模型版本 | 出价规则 |
|---|---|---|---|---|
| S100238 | 高价值加购召回 | 加购未购 | CVR_V3 | 动态系数 |

---

## 7.2 策略 ID 要和模型版本、实验版本解耦

不建议每次模型小版本变更都换策略 ID，否则报表会被切碎。

较好的方式：

```text
strategy_id = S_CART_RECALL
model_version = CVR_V3
experiment_id = EXP_20260623_A
```

这样更利于分析：

- 策略长期效果
- 模型版本效果
- 实验组效果

---

## 7.3 策略 ID 要支持生命周期管理

每个策略 ID 应该有状态：

| 状态 | 含义 |
|---|---|
| draft | 草稿 |
| testing | 测试中 |
| online | 生效中 |
| paused | 暂停 |
| expired | 已过期 |
| offline | 已下线 |

避免已下线策略继续被 RTA 服务返回。

# 8. 简化示例

## 8.1 策略配置

```text
策略 ID：S_CART_7D_HIGH
策略名称：近 7 天加购未购高转化用户
命中条件：
- 近 7 天有加购行为
- 未完成购买
- pCVR > 0.06
出价规则：
- bid_coefficient = 1.6
```

## 8.2 RTA 请求

```json
{
  "request_id": "req_10001",
  "media_user_id": "m_abc",
  "slot_id": "feed_001"
}
```

## 8.3 广告主侧判断

```text
ID Mapping 成功
↓
识别为广告主用户 u_123
↓
近 7 天加购过商品
↓
未购买
↓
pCVR = 0.083
↓
命中策略 S_CART_7D_HIGH
```

## 8.4 RTA 返回

```json
{
  "request_id": "req_10001",
  "accept": true,
  "strategy_id": "S_CART_7D_HIGH",
  "bid_coefficient": 1.6
}
```

## 8.5 媒体侧处理

```text
基础出价 = 10 元 CPM
出价系数 = 1.6
广告主有效出价 = 16 元 CPM
再结合媒体侧 pCTR、质量分、底价、竞价环境进行排序
```

# 9. 一句话总结

**策略 ID 在媒体侧主要用于识别广告主命中的实时投放策略，进而支撑出价融合、计划映射、预算频控、实验分组、归因分析和问题排查；它通常由广告主策略平台在策略创建时生成，并在 RTA 请求阶段由广告主 RTA 服务根据 ID Mapping、用户特征、模型分数和业务规则实时匹配后返回。**
