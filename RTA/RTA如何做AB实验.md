# 外投人群包 vs RTA实时预估：AB实验怎么做，是否需要联合媒体侧，和策略ID什么关系

## 1. 核心结论

| 问题 | 结论 |
|---|---|
| **外投人群包能做AB吗？** | 能做，通常通过**广告主侧人群分桶 + 上传不同人群包 + 媒体侧绑定不同计划/单元**实现。 |
| **RTA实时预估能做AB吗？** | 能做，通常通过**RTA请求实时分桶 + 返回不同参竞/出价系数策略 + 回传实验ID/策略ID**实现。 |
| **是否需要联合媒体侧？** | **强烈建议联合媒体侧**。人群包AB需要媒体侧支持计划隔离、预算隔离、流量稳定；RTA AB更依赖媒体侧透传实验字段、策略ID、RTA ID、消耗与转化数据拆分。 |
| **和策略ID有关吗？** | **有关，但策略ID不是AB实验本身**。策略ID用于标识一套投放/RTA决策逻辑，AB实验需要在策略ID、实验ID、计划ID、token之间建立清晰映射。 |
| **最关键原则** | AB实验必须做到：**随机分桶、互斥隔离、口径一致、预算可比、归因可拆、实验ID可追踪**。 |

---

# 2. 两种模式的本质差异

## 2.1 外投人群包模式

外投人群包模式下，广告主先在站内对存量用户做价值预估，例如对10亿用户预测 **CTCVR、GMV、LTV、复购概率、流失概率** 等，然后离线圈选用户，生成不同价值层级的人群包，再上传到腾讯、字节等媒体平台。

媒体侧请求广告时，不会实时调用广告主模型，而是判断当前用户是否命中已上传的人群包。命中则参与对应广告计划的投放，不命中则过滤。这个模式的特点是：**决策前置、更新滞后、实验主要依赖人群包和广告计划隔离**。

---

## 2.2 RTA实时预估模式

RTA模式下，媒体在广告竞价前通过实时接口请求广告主，广告主结合第一方数据和实时模型判断是否参竞，并可返回 **是否参竞、出价系数、实验ID** 等信息。公开资料中提到，RTA会由媒体实时请求广告主，广告主返回参竞结果、出价系数、实验ID等字段；出价系数通常会影响本次请求的最终出价，例如“后台出价 × 出价系数”。[〔1〕](https://xiaomi.juxuan.net/xiaomi/zx/778.html)

RTA模式的特点是：**决策实时、策略灵活、可做精细分桶、可动态出价，但对低延迟、高并发、日志追踪要求更高**。资料中也提到，RTA通常要求广告主在几十毫秒内返回结果，例如小米RTA要求50ms内完成响应，百度相关资料中也提到广告主需要在60ms内回复是否选择该流量。[〔1〕](https://xiaomi.juxuan.net/xiaomi/zx/778.html)[〔3〕](https://www.fxbaogao.com/detail/5409296)

---

# 3. 外投人群包模式下如何做AB实验

## 3.1 实验目标先定清楚

人群包AB通常适合验证：

1. **人群选择策略**
   - 高CTCVR人群 vs 高LTV人群；
   - 高点击高转化人群 vs 高客单价人群；
   - 老客召回人群 vs 流失风险人群。

2. **阈值策略**
   - Top 5%人群 vs Top 10%人群；
   - CTCVR > 0.8‰ vs CTCVR > 1.2‰；
   - 只投高价值人群 vs 高中价值人群都投。

3. **媒体组合策略**
   - 腾讯同一人群 vs 字节同一人群；
   - 同一媒体下不同版位；
   - 同一人群在不同广告产品形态下的表现。

4. **出价/预算策略**
   - 同一人群不同出价；
   - 同一人群不同预算节奏；
   - 同一人群不同oCPX目标成本。

---

## 3.2 标准做法：广告主侧先随机分桶

以10亿站内用户为例，不建议直接按模型分排序后切一半做A、一半做B，因为这样两组用户质量天然不同。应该先做**随机分桶**。

### 推荐流程

```text
用户全集 U
→ 过滤不可投用户
→ 按 user_id/device_id 做稳定hash
→ 分为 A桶、B桶、Holdout桶
→ 在每个桶内分别应用对应策略
→ 生成 A人群包、B人群包
→ 上传媒体
→ 媒体侧绑定不同广告计划/单元
→ 分别统计曝光、点击、转化、GMV、ROI
```

### 示例

| 分组 | 分桶逻辑 | 策略 | 上传媒体后的使用方式 |
|---|---|---|---|
| A组 | hash(user_id) % 100 < 45 | CTCVR Top 10% | 绑定计划A |
| B组 | 45 ≤ hash(user_id) % 100 < 90 | CTCVR Top 15% + LTV约束 | 绑定计划B |
| Holdout | 90 ≤ hash(user_id) % 100 < 100 | 不投或低曝光 | 用于衡量自然转化基线 |

**Holdout组很重要**。如果只比较A、B投放效果，容易看不清广告真实增量；加入Holdout后，可以评估广告带来的**增量转化、增量GMV、增量ROI**。

---

## 3.3 媒体侧需要配合什么

人群包AB虽然主要在广告主侧做分桶，但**最好联合媒体侧**，否则实验容易失真。

### 媒体侧需要做的事情

| 媒体侧事项 | 目的 |
|---|---|
| **分别绑定A/B人群包到不同计划或单元** | 确保数据可拆分 |
| **A/B计划预算尽量一致** | 避免一组预算不足导致曝光不足 |
| **出价方式一致** | 避免算法学习目标不同 |
| **投放时段、地域、版位一致** | 减少外部变量 |
| **排除A/B人群重叠** | 避免同一用户同时进入两组 |
| **保持创意一致或明确创意变量** | 避免创意差异污染人群策略实验 |
| **提供计划/单元维度消耗、曝光、点击、转化数据** | 支持实验评估 |

如果媒体侧不配合，也可以通过广告主侧归因数据回收做评估，但会有明显问题：**无法保证流量均匀、无法控制媒体算法学习差异、无法稳定隔离预算与频控**。

---

## 3.4 人群包AB的典型实验设计

### 实验一：验证模型版本

| 项目 | A组 | B组 |
|---|---|---|
| 模型 | 老CTCVR模型 | 新CTCVR模型 |
| 圈选方式 | Top 10% | Top 10% |
| 出价 | 相同 |
| 创意 | 相同 |
| 预算 | 相同 |
| 观察指标 | CPA、CVR、GMV、ROI、增量转化 |

核心判断：

```text
如果 B组ROI更高，且CPA不劣于A组，新模型胜出；
如果 B组消耗不足，需要判断是人群规模太小还是媒体学习不足；
如果 B组CVR高但GMV低，说明模型优化目标可能偏前链路。
```

---

### 实验二：验证圈选阈值

| 项目 | A组 | B组 |
|---|---|---|
| 人群 | CTCVR Top 5% | CTCVR Top 15% |
| 目标 | 高质量窄人群 | 扩量人群 |
| 重点观察 | ROI、CPA | 消耗、GMV、边际ROI |

核心判断：

```text
A组ROI高但消耗小，适合保ROI；
B组ROI略低但GMV更高，适合扩量；
如果B组边际ROI仍达标，可以继续放宽阈值。
```

---

# 4. RTA实时预估模式下如何做AB实验

## 4.1 RTA AB的核心优势

RTA可以在每一次广告请求上实时决策，适合做更细粒度的实验：

1. **模型版本AB**
   - 老CTCVR模型 vs 新CTCVR模型；
   - 单目标模型 vs 多目标模型；
   - CTCVR模型 vs CTCVR × LTV模型。

2. **参竞阈值AB**
   - 阈值0.001参竞 vs 阈值0.0015参竞；
   - 固定阈值 vs 分媒体/分版位动态阈值。

3. **出价系数AB**
   - 分档系数 vs 连续函数系数；
   - 保守出价 vs 激进出价；
   - ROI约束出价 vs GMV扩量出价。

4. **策略组合AB**
   - 拉新策略 vs 召回策略；
   - 高频召回过滤 vs 不过滤；
   - 高LTV用户提价 vs 不提价。

百度资料中提到，RTA可以支持A/B实验，即平台将流量分为不同实验组，广告主可以按不同实验分组返回不同策略，并通过实验数据衡量策略优劣。[〔3〕](https://www.fxbaogao.com/detail/5409296)

---

## 4.2 RTA AB的推荐流程

```text
媒体广告请求
→ 媒体携带设备ID、广告位、计划、RTA token等信息请求广告主
→ 广告主RTA网关接收请求
→ 根据稳定hash或媒体实验分桶字段分配A/B组
→ 调用不同模型/策略
→ 返回是否参竞、出价系数、实验ID/策略ID
→ 媒体侧结合返回结果参与竞价
→ 广告曝光/点击/转化
→ 双方按实验ID/策略ID/计划ID拆分报表
→ 分析CPA、ROI、GMV、消耗、通过率、竞胜率、超时率
```

RTA返回通常包括是否参与竞价、出价系数、实验ID等信息；腾讯、字节、百度、小米等媒体的字段命名和接口细节会不同，但决策逻辑大体类似：广告主实时返回“参不参竞”和“该给多大价值信号”。[〔1〕](https://xiaomi.juxuan.net/xiaomi/zx/778.html)[〔3〕](https://www.fxbaogao.com/detail/5409296)

---

## 4.3 RTA AB分桶方式

### 方式一：广告主侧分桶

广告主收到RTA请求后，使用用户ID、设备ID或媒体侧唯一请求标识做稳定hash。

```text
bucket = hash(media_id + device_id + experiment_salt) % 100

0-49：A组
50-99：B组
```

优点：

- 广告主可控；
- 不强依赖媒体实验系统；
- 适合快速验证模型和策略。

缺点：

- 媒体侧如果无法识别实验ID，报表拆分较困难；
- 媒体算法学习可能仍混在同一计划里，导致A/B相互影响；
- 如果A/B返回不同通过率，媒体侧实际曝光分布可能不均衡。

---

### 方式二：媒体侧分桶

由媒体侧实验平台先对流量分桶，RTA请求中带上实验分组，广告主按分组返回不同策略。

优点：

- 流量分配更规范；
- 媒体侧报表更容易拆分；
- 可以减少媒体算法学习层面的串扰；
- 更适合正式实验或大预算实验。

缺点：

- 需要媒体侧排期和支持；
- 接入成本更高；
- 不同媒体实验能力和字段支持不同。

---

### 方式三：计划/单元隔离分桶

在媒体侧建两个计划或两个广告组：

| 实验组 | 媒体计划 | RTA token | 策略 |
|---|---|---|---|
| A组 | 计划A | token_A | 老模型 |
| B组 | 计划B | token_B | 新模型 |

优点：

- 数据拆分清晰；
- 预算、出价、创意可以独立控制；
- 对媒体报表友好。

缺点：

- 媒体算法学习会分别进行，可能带来学习差异；
- 两个计划可能互相抢量；
- 需要控制预算和出价一致；
- 对实验设计要求较高。

---

## 4.4 RTA AB中如何返回出价系数

RTA实时预估常见返回逻辑是：

```text
如果 CTCVR < 参竞阈值：不参竞
如果 CTCVR ≥ 参竞阈值：参竞，并根据价值分层返回出价系数
```

### 示例一：分档系数

| CTCVR分位 | 用户价值 | 是否参竞 | 出价系数 |
|---|---|---|---|
| P0-P50 | 低价值 | 否 | 0 |
| P50-P70 | 中低价值 | 是 | 0.7 |
| P70-P90 | 中高价值 | 是 | 1.0 |
| P90-P97 | 高价值 | 是 | 1.3 |
| P97-P100 | 极高价值 | 是 | 1.6 |

### 示例二：连续函数系数

```text
bid_coef = clip(CTCVR / baseline_CTCVR, min_coef, max_coef)
```

例如：

```text
baseline_CTCVR = 当前媒体/广告位平均CTCVR
min_coef = 0.5
max_coef = 2.0

当用户CTCVR是均值的1.5倍时，bid_coef = 1.5
当用户CTCVR是均值的0.4倍时，bid_coef = 0.5或不参竞
```

### 示例三：ROI约束系数

如果广告主目标是ROI，而不是单纯转化：

```text
expected_value = pCVR × expected_GMV × expected_margin
max_cost = expected_value / target_ROI
bid_coef = max_cost / media_base_bid
```

即：

```text
出价系数 = 本次流量广告主可接受成本 / 媒体侧基础出价
```

这种方式比简单按CTCVR分档更适合电商，因为高CTCVR用户不一定高GMV，高GMV用户也不一定高ROI。

# 5. RTA AB是否需要联合媒体侧？

## 5.1 简短回答：从“能做”到“做准”，必须联合媒体侧

RTA AB理论上广告主自己就能做，但如果希望实验结果可信，建议联合媒体侧。原因是RTA处在竞价链路中，广告主返回只是第一步，后续还会经过媒体侧的预估、排序、竞价、频控、预算分配、创意优选。

RTA不是单纯接口能力，而是广告主在广告曝光前实时决定这次流量是否值得投。公开资料也指出，RTA是广告平台在流量到来时，通过实时接口询问广告主是否要这次流量，广告主基于一方数据返回决策结果。[〔7〕](https://blog.csdn.net/zhangzehai2234/article/details/149209997)[〔9〕](https://blog.csdn.net/sanjie333/article/details/148725032)

---

## 5.2 哪些实验必须联合媒体侧

| 实验类型 | 是否必须联合媒体侧 | 原因 |
|---|---|---|
| 模型离线AB | 不必须 | 可以在广告主日志中回放评估 |
| RTA参竞阈值小流量灰度 | 建议 | 需要观察媒体通过率、竞胜率、消耗 |
| 出价系数AB | **必须强配合** | 出价会影响媒体竞价排序和成本 |
| 计划级预算AB | **必须** | 媒体侧控制预算和投放节奏 |
| 版位/流量包AB | **必须** | 需要媒体侧拆流量 |
| 增量实验/Holdout | **强烈建议** | 需要排除曝光或控制触达 |
| 跨媒体对比 | 建议 | 需要统一归因、监测、口径 |

---

## 5.3 媒体侧需要支持哪些能力

| 能力 | 作用 |
|---|---|
| **RTA token/策略绑定** | 区分不同RTA策略 |
| **实验ID透传或回传** | 用于A/B报表拆分 |
| **RTA ID或请求ID追踪** | 串联请求、返回、曝光、点击、转化 |
| **计划/广告组隔离** | 控制预算、出价、创意、流量 |
| **分媒体、分广告位报表** | 排查不同流量来源差异 |
| **通过率、超时率、竞胜率监控** | 判断RTA策略是否健康 |
| **归因数据回传** | 评估转化、GMV、ROI |
| **频控/排重能力** | 避免A/B用户重复触达 |

# 6. 和策略ID有关吗？

## 6.1 策略ID、实验ID、token、计划ID的关系

这几个ID经常混用，但含义不同。

| 标识 | 所属方 | 主要作用 | 是否等同AB实验 |
|---|---|---|---|
| **计划ID / 广告组ID** | 媒体侧 | 控制预算、出价、创意、投放范围 | 不是，但可用于实验隔离 |
| **人群包ID** | 广告主/媒体侧 | 标识上传的人群包 | 不是，但可对应实验组 |
| **RTA token** | 媒体侧/广告主共同约定 | 标识RTA接入、投放策略或场景，如拉新/拉活 | 不是，但常用于策略隔离 |
| **策略ID strategy_id** | 广告主侧 | 标识一套决策逻辑、模型版本、阈值、出价函数 | 不是AB本身，但必须记录 |
| **实验ID experiment_id** | 广告主或媒体实验平台 | 标识AB实验及实验组 | 是AB分析核心字段 |
| **RTA ID / request_id** | 媒体侧或双方 | 串联单次请求、返回、曝光、点击、转化 | 不是AB，但用于追踪链路 |

小米相关资料中提到，RTA token可表示广告主投放策略，例如拉新、拉活；广告主返回结果中也可以包含实验ID。[〔1〕](https://xiaomi.juxuan.net/xiaomi/zx/778.html)

---

## 6.2 推荐映射方式

建议建立一张实验配置表：

| experiment_id | group | media | plan_id | rta_token | strategy_id | model_version | threshold | bid_coef_func |
|---|---|---|---|---|---|---|---|---|
| exp_20260624_001 | A | 腾讯 | plan_A | token_A | strategy_v1 | model_v10 | 0.0010 | coef_v1 |
| exp_20260624_001 | B | 腾讯 | plan_B | token_B | strategy_v2 | model_v11 | 0.0010 | coef_v1 |
| exp_20260624_002 | A | 字节 | plan_A | token_A | strategy_v3 | model_v10 | 0.0012 | coef_v1 |
| exp_20260624_002 | B | 字节 | plan_B | token_B | strategy_v4 | model_v10 | 0.0012 | coef_v2 |

这样才能做到：

```text
一次请求 → 属于哪个媒体 → 哪个计划 → 哪个token → 哪个实验 → 哪个实验组 → 哪个策略 → 哪个模型 → 返回了什么 → 最后是否曝光/点击/转化
```

# 7. 人群包AB vs RTA AB：完整对比

| 维度 | 外投人群包AB | RTA实时预估AB |
|---|---|---|
| 决策时点 | 投放前离线圈人 | 广告请求时实时决策 |
| 分桶方式 | 广告主侧用户hash分桶，人群包隔离 | 广告主侧hash、媒体侧实验分桶、计划/token隔离 |
| 实验载体 | 人群包ID、计划ID | RTA token、strategy_id、experiment_id、request_id |
| 策略变化频率 | 低，通常天级更新 | 高，可分钟级或小时级灰度 |
| 出价能力 | 主要靠媒体计划出价/oCPX | 可返回出价系数、价值分 |
| 是否需媒体配合 | 建议 | 强烈建议，出价AB基本必须 |
| 主要风险 | 人群包重叠、更新滞后、计划学习差异 | 超时、通过率异常、竞胜率波动、实验组串扰 |
| 适合验证 | 人群圈选、模型排序、媒体组合 | 实时模型、阈值、出价系数、动态策略 |
| 评估指标 | CPA、CVR、ROI、GMV、增量转化 | 通过率、竞胜率、eCPM、CPA、ROI、GMV、超时率 |

---

# 8. 推荐实验方案

## 8.1 如果当前还在人群包阶段

建议采用：

```text
广告主侧稳定hash分桶
+ A/B人群包互斥
+ 媒体侧计划隔离
+ 预算/出价/创意一致
+ Holdout增量组
```

### 最小可行实验设计

| 组别 | 策略 | 占比 | 操作 |
|---|---|---|---|
| A组 | 老模型Top 10% | 45% | 上传A人群包，绑定计划A |
| B组 | 新模型Top 10% | 45% | 上传B人群包，绑定计划B |
| Holdout | 不投 | 10% | 不上传或上传排除包 |

观察：

```text
B组 ROI 是否显著高于 A组
B组 CPA 是否不高于 A组
B组 GMV 是否提升
B组 相比 Holdout 是否有更高增量
```

---

## 8.2 如果已经升级到RTA实时预估

建议采用：

```text
小流量灰度
→ 媒体/计划/token隔离
→ RTA实时返回experiment_id和strategy_id
→ 监控通过率、超时率、竞胜率、成本、ROI
→ 达标后逐步放量
```

### 最小可行实验设计

| 组别 | 策略 | 流量占比 | 返回内容 |
|---|---|---|---|
| A组 | 老CTCVR模型 + 老出价系数 | 45% | is_bid、bid_coef、experiment_id=A、strategy_id=v1 |
| B组 | 新CTCVR模型 + 新出价系数 | 45% | is_bid、bid_coef、experiment_id=B、strategy_id=v2 |
| Holdout | 不参竞或极低比例参竞 | 10% | is_bid=0 或单独标识 |

核心监控：

| 指标 | 说明 |
|---|---|
| **RTA请求量** | 两组流量是否均衡 |
| **通过率** | 返回参竞比例 |
| **超时率** | 是否影响媒体接入质量 |
| **竞胜率** | 参竞后是否赢得曝光 |
| **曝光/点击/转化** | 后链路表现 |
| **CPA/ROI/GMV** | 最终业务效果 |
| **分媒体/分版位表现** | 判断策略是否局部有效 |
| **出价系数分布** | 防止过度提价或过度压价 |

# 9. 常见坑与规避方法

## 9.1 A/B用户不互斥

如果同一用户同时进入A组和B组，实验结果会被污染。

**规避：**

```text
统一用 user_id/device_id 做稳定hash
实验周期内不变更分桶salt
人群包上传前做交集检查
RTA请求按同一套分桶服务决策
```

---

## 9.2 只看转化率，不看消耗和竞胜率

RTA B组可能转化率高，但通过率太低、竞胜率太低，导致消耗不足，无法规模化。

**规避：**

同时看：

```text
通过率 → 竞胜率 → 曝光量 → 点击率 → 转化率 → CPA → ROI → GMV
```

---

## 9.3 出价系数AB没有控制媒体计划差异

如果A组和B组使用不同预算、不同出价、不同创意，结果无法判断是RTA策略导致，还是媒体计划差异导致。

**规避：**

尽量保证：

```text
同媒体
同版位
同预算
同出价目标
同创意
同投放时段
同归因口径
```

---

## 9.4 RTA返回了实验ID，但曝光/转化数据无法回连

这是非常常见的问题。只在RTA日志里记录实验ID不够，还要能和媒体曝光、点击、转化日志关联。

**规避：**

要求链路中保留：

```text
request_id / rta_id
experiment_id
strategy_id
token
plan_id
device_id加密标识
timestamp
is_bid
bid_coef
exposure_id
click_id
conversion_id
```

---

## 9.5 媒体侧缓存影响实验

部分RTA系统可能存在缓存或“小池子”机制。如果媒体侧不是每次都实时请求广告主，而是读取历史缓存结果，实验分组可能不准确。公开资料中提到，若广告主不能在规定时间内返回，部分RTA会使用上一次缓存结果，查不到则默认不参竞，这会导致通过率口径不一致。[〔1〕](https://xiaomi.juxuan.net/xiaomi/zx/778.html)

**规避：**

- 实验前确认媒体侧是否实时请求；
- 缓存key是否包含实验ID或策略ID；
- A/B策略切换时是否清缓存；
- 分析时区分实时请求流量和缓存流量。

---

# 10. 最终建议

## 10.1 人群包模式

优先使用：

```text
广告主侧随机分桶
+ 人群包互斥
+ 媒体计划隔离
+ Holdout增量组
```

适合验证：**模型排序、人群阈值、人群扩量、媒体组合**。

---

## 10.2 RTA实时预估模式

优先使用：

```text
媒体侧实验分桶或计划/token隔离
+ 广告主侧返回experiment_id、strategy_id、bid_coef
+ 请求到转化全链路日志打通
+ 小流量灰度后逐步放量
```

适合验证：**实时模型、参竞阈值、出价系数、ROI约束、动态策略**。

---

## 10.3 是否需要联合媒体侧

**结论：需要。**

- 人群包AB：可以广告主侧主导，但媒体侧需要配合计划隔离和报表拆分。
- RTA AB：建议广告主和媒体联合设计，尤其是涉及**出价系数、竞价排序、预算分配、实验ID回传**时，必须联合媒体侧。

---

## 10.4 和策略ID的关系

**策略ID不是AB实验本身，但它是AB实验可追踪、可复盘、可解释的关键字段。**

推荐关系：

```text
experiment_id：标识一次AB实验
group_id：标识A/B/C组
strategy_id：标识具体策略逻辑
model_version：标识模型版本
rta_token：标识媒体侧RTA投放场景
plan_id：标识媒体计划
request_id/rta_id：标识单次请求
```

只要这套ID体系设计清楚，外投人群包和RTA实时预估都可以做严谨AB；区别在于：**人群包AB更偏离线人群实验，RTA AB更偏实时竞价策略实验**。
