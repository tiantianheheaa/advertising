- 问题：从人群包升级为RTA实时预估，是否参竞、出价系数如何给出？

# 外投RTA中：如何用实时CTCVR判断参竞与给出出价系数

下面从 **核心公式、参竞判断、出价系数生成、全流程、腾讯/字节等头部媒体差异** 五部分说明。重点结论是：

> **实时RTA不是简单“CTCVR高就投、低就不投”，而是要把广告主侧CTCVR转成“本次曝光的可接受成本”或“相对媒体基准价值”，再结合预算、成本、ROI、频控、库存、用户状态等约束，决定是否参竞以及返回多大的出价系数。**

---

## 1. 基础理解：广告主侧实时CTCVR是什么

在外投RTA中，广告主侧实时模型通常预估的是：

```text
CTCVR_adv = P(用户在本次广告曝光后最终发生目标转化 | 用户、媒体、场景、商品、时间等特征)
```

如果拆开理解：

```text
CTCVR = CTR × CVR
```

即：

```text
P(conversion | impression)
= P(click | impression) × P(conversion | click, impression)
```

在推荐、广告系统中，CTCVR通常表示从曝光到转化的整体概率，pCTCVR可由pCTR和pCVR相乘得到。[〔5〕](https://blog.csdn.net/livan1234/article/details/85159693)

在RTA场景里，媒体侧通常掌握更完整的 **曝光场景、广告位、实时点击行为、平台内用户行为**，广告主侧掌握更强的 **站内交易、会员、品类偏好、生命周期、LTV、复购、价格敏感度、流失风险** 等数据。因此，RTA的价值在于：媒体来请求时，广告主实时判断这个用户对自己是否值得买。RTA下广告主可以向媒体返回是否参竞、价值信息或出价调节系数，媒体再结合自身预估完成竞价排序。公开资料中也提到，RTA联合出价可理解为 `bid = cpa × pctr × pcvr × tuning_bid_adv`，其中 `tuning_bid_adv` 由广告主负责，用于表达广告主后端转化价值或修正媒体侧预估。[〔1〕](https://zhuanlan.zhihu.com/p/385993960)

---

# 2. 如何根据实时CTCVR判断是否参竞

## 2.1 第一层：硬规则过滤

在模型分数之前，应先做 **业务与合规硬过滤**。常见包括：

| 过滤项 | 判断逻辑 | 示例 |
|---|---|---|
| 用户是否可识别 | 媒体请求ID是否能映射到广告主侧用户或设备 | 无法识别则走默认策略或不参竞 |
| 是否符合合规授权 | 是否满足用户授权、数据使用范围、媒体协议 | 不满足则不参竞 |
| 用户状态 | 是否已购买、已转化、已退订、黑名单、风险用户 | 已完成目标且不需复购则不参竞 |
| 商品/活动状态 | 商品是否有库存、价格是否有效、活动是否在线 | 无库存不参竞 |
| 频控 | 最近是否已曝光过多、点击过多、转化过 | 超频则不参竞 |
| 预算 | 计划、单元、活动预算是否充足 | 预算不足则不参竞 |
| 投放窗口 | 是否在允许的时段、地域、媒体、广告位内 | 不符合则不参竞 |

这一层的目标是避免模型高分但 **业务上不能投、投了浪费、投了有风险** 的情况。

---

## 2.2 第二层：基于CTCVR阈值判断

最简单的参竞规则是：

```text
如果 CTCVR_adv ≥ 阈值 τ，则参竞
如果 CTCVR_adv < 阈值 τ，则不参竞
```

但实际不能用一个全局固定阈值，因为腾讯、字节、不同广告位、不同商品、不同转化目标的流量质量和成本差异很大。更合理的是分层阈值：

```text
τ = f(媒体, 广告位, 转化目标, 商品类目, 用户生命周期, 时段, 预算状态)
```

例如：

| 场景 | 阈值策略 |
|---|---|
| 腾讯视频信息流 | 流量较泛，阈值可设高一些 |
| 微信生态高意图召回 | 用户价值明确，可适当降低阈值 |
| 抖音短视频推荐流 | 用户兴趣变化快，需结合实时内容场景动态调阈值 |
| 高客单价商品 | CTCVR低但客单价值高，不能只看转化率 |
| 低客单价高频商品 | 更依赖CTCVR阈值和成本控制 |
| 拉新目标 | 阈值可低一些，保留探索 |
| 复购/召回目标 | 阈值应更严格，强调ROI |

---

## 2.3 第三层：用“预期价值”判断是否值得买

仅看CTCVR不够，因为不同用户转化后的价值不同。更合理的方式是计算 **本次曝光的预期转化价值**。

### 方式一：CPA目标下的参竞判断

如果广告主目标是控制CPA，例如目标CPA为80元：

```text
本次曝光可接受成本 = CTCVR_adv × 目标CPA
```

换算成eCPM：

```text
可接受eCPM = CTCVR_adv × 目标CPA × 1000
```

然后与预估市场胜出成本比较：

```text
如果 可接受eCPM ≥ 预估胜出eCPM × 安全系数，则参竞
否则不参竞
```

### 示例

假设：

```text
目标CPA = 80元
用户A的实时CTCVR = 0.08% = 0.0008
用户B的实时CTCVR = 0.02% = 0.0002
```

则：

```text
用户A可接受eCPM = 0.0008 × 80 × 1000 = 64元
用户B可接受eCPM = 0.0002 × 80 × 1000 = 16元
```

如果当前媒体广告位的预估胜出eCPM约为45元：

| 用户 | CTCVR | 可接受eCPM | 是否参竞 |
|---|---:|---:|---|
| A | 0.08% | 64元 | 参竞 |
| B | 0.02% | 16元 | 不参竞 |

---

### 方式二：ROI / ROAS目标下的参竞判断

如果广告主目标是ROI或ROAS，则需要引入预估GMV、毛利或LTV。

```text
预期GMV = CTCVR_adv × 预估客单价
```

如果目标ROAS为5：

```text
本次曝光可接受成本 = 预期GMV / 目标ROAS
```

换算成eCPM：

```text
可接受eCPM = CTCVR_adv × 预估客单价 / 目标ROAS × 1000
```

### 示例

假设：

```text
用户实时CTCVR = 0.1% = 0.001
预估客单价 = 200元
目标ROAS = 5
```

则：

```text
本次曝光可接受成本 = 0.001 × 200 / 5 = 0.04元
可接受eCPM = 0.04 × 1000 = 40元
```

如果该媒体场景预估胜出eCPM低于40元，可以参竞；如果明显高于40元，应降低系数或不参竞。

---

## 2.4 第四层：结合预算节奏和成本反馈

实际线上不能只按单次请求判断，还要看全局投放状态。

常见控制逻辑：

```text
最终参竞 = 硬规则通过
        AND CTCVR达到阈值
        AND 预期价值覆盖成本
        AND 预算节奏允许
        AND 当前CPA/ROI未明显恶化
```

例如：

| 状态 | 策略 |
|---|---|
| 当前CPA低于目标，消耗不足 | 适当降低参竞阈值，提高系数 |
| 当前CPA高于目标，消耗正常 | 提高阈值，降低系数 |
| 当前ROI低于目标 | 只保留高CTCVR、高客单、高LTV人群 |
| 预算消耗过快 | 提高阈值，压低系数 |
| 预算消耗过慢 | 放宽阈值，增加探索流量 |
| 新计划冷启动 | 保留小比例探索，不完全依赖高分流量 |

---

# 3. 出价系数如何给出

RTA中常见有两种模式：

1. **联合出价 / 系数模式**：广告主返回 `bid_factor`、`tuning_bid_adv`、价值分层等，由媒体侧结合自身pCTR、pCVR、质量分、广告主出价完成竞价。
2. **广告主直接出价模式**：广告主直接返回本次请求愿意出的价格，例如bid或eCPM。但这种模式对广告主侧实时点击率、市场价格、媒体排序机制要求更高，实际落地难度更大。资料中也提到，广告主直接出价类似RTB通过DSP出价，但如果广告主拿不到实时点击相关信息，缺少pCTR时直接出价会比较困难。[〔1〕](https://zhuanlan.zhihu.com/p/385993960)

头部媒体RTA更常见的是 **联合出价/系数模式**。

---

## 3.1 系数模式的基本思想

媒体侧基础排序可抽象为：

```text
媒体基础bid = 目标CPA × pCTR_media × pCVR_media
```

RTA加入广告主侧价值调节后：

```text
最终bid = 目标CPA × pCTR_media × pCVR_media × bid_factor_adv
```

其中：

```text
bid_factor_adv = 广告主返回的出价系数
```

该系数表达的是：

> **广告主认为这个用户相对于媒体平均预估，是更值得买，还是不值得买。**

例如：

| 广告主判断 | bid_factor |
|---|---:|
| 明确高价值用户 | 2.0 - 3.0 |
| 中高价值用户 | 1.2 - 2.0 |
| 普通用户 | 0.8 - 1.2 |
| 低价值用户 | 0.2 - 0.8 |
| 不值得买 | no_bid / 0 |

---

## 3.2 基于CTCVR相对值生成系数

最常用的方法是把用户实时CTCVR与某个基准CTCVR比较。

```text
raw_factor = CTCVR_adv / CTCVR_base
```

其中：

```text
CTCVR_base = 当前广告计划、媒体、广告位、商品类目下的平均CTCVR
```

例如：

```text
当前计划在字节信息流上的平均CTCVR = 0.04%
某用户实时CTCVR = 0.08%
```

则：

```text
raw_factor = 0.08% / 0.04% = 2.0
```

说明该用户价值约为平均水平的2倍，可以返回较高系数。

如果：

```text
某用户实时CTCVR = 0.015%
```

则：

```text
raw_factor = 0.015% / 0.04% = 0.375
```

可以低系数参竞，或者直接不参竞。

---

## 3.3 加入客单价、毛利、LTV后的价值系数

对于电商广告主，不能只看CTCVR。高价值用户可能转化率不最高，但客单价、毛利、复购或会员价值更高。

可以定义：

```text
用户预期价值 = CTCVR_adv × 预估客单价 × 毛利率 × LTV系数
```

再与基准价值比较：

```text
raw_factor = 用户预期价值 / 基准用户预期价值
```

例如：

| 用户 | CTCVR | 预估客单价 | 预期GMV |
|---|---:|---:|---:|
| A | 0.10% | 80元 | 0.08元 |
| B | 0.05% | 300元 | 0.15元 |

虽然B的CTCVR只有A的一半，但预期GMV更高，B反而可以返回更高系数。

---

## 3.4 加入成本反馈后的最终系数

线上最终系数通常不是模型分数直接映射，而是：

```text
最终bid_factor =
clip(
  价值系数
  × 成本反馈系数
  × 预算节奏系数
  × 媒体校准系数
  × 探索系数,
  最小系数,
  最大系数
)
```

其中：

| 系数 | 作用 |
|---|---|
| 价值系数 | 由CTCVR、GMV、LTV决定 |
| 成本反馈系数 | 当前CPA高于目标则降低，低于目标则提高 |
| 预算节奏系数 | 消耗慢则提高，消耗快则降低 |
| 媒体校准系数 | 修正腾讯、字节等媒体流量分布差异 |
| 探索系数 | 给新用户、新商品、新场景保留小流量探索 |
| clip上下限 | 防止系数过大导致成本失控，或过小导致无消耗 |

示例：

```text
价值系数 = 2.0
成本反馈系数 = 0.9
预算节奏系数 = 1.1
媒体校准系数 = 1.0
探索系数 = 1.0
```

则：

```text
最终bid_factor = 2.0 × 0.9 × 1.1 × 1.0 × 1.0 = 1.98
```

如果媒体允许系数范围为 `[0.2, 3.0]`，则返回：

```text
bid_factor = 1.98
```

---

## 3.5 分桶映射：线上更稳的做法

实际生产中，很多广告主不会直接返回连续系数，而是把CTCVR或价值分映射到分桶。

示例：

| CTCVR分位 | 用户价值 | 参竞策略 | bid_factor |
|---|---|---|---:|
| Top 1% | 极高价值 | 强参竞 | 3.0 |
| Top 5% | 高价值 | 强参竞 | 2.0 |
| Top 20% | 中高价值 | 正常参竞 | 1.2 |
| Top 50% | 普通价值 | 低系数参竞 | 0.6 |
| Bottom 50% | 低价值 | 不参竞或探索 | 0 / 0.2 |

分桶的优点：

- 稳定；
- 易解释；
- 方便和媒体联调；
- 降低模型分数抖动对出价的影响；
- 便于做AB实验和成本控制。

---

# 4. 实时RTA全流程：从媒体请求到广告展示

## 4.1 全流程

```text
1. 用户打开腾讯/字节等媒体App
        ↓
2. 媒体产生广告请求
        ↓
3. 媒体侧完成基础召回和广告候选过滤
        ↓
4. 媒体向广告主RTA服务发起请求
        ↓
5. 广告主识别用户、广告计划、商品、场景
        ↓
6. 广告主实时特征查询
        ↓
7. 广告主CTCVR / GMV / LTV模型实时预估
        ↓
8. 广告主判断是否参竞
        ↓
9. 广告主返回 no_bid 或 bid_factor / value_score
        ↓
10. 媒体侧结合pCTR、pCVR、广告主出价、质量分进行排序竞价
        ↓
11. 胜出后广告展示
        ↓
12. 用户点击、访问落地页、加购、下单、支付
        ↓
13. 广告主向媒体回传转化数据
        ↓
14. 双方模型更新：媒体优化pCTR/pCVR，广告主优化RTA模型和系数策略
```

传统人群包方式存在一个问题：广告主提前上传人群包后，用户是否参与竞价、何时展示、是否胜出都不完全可控，且难以在特定时间召回特定用户；RTA通过实时请求与响应，可以在媒体请求发生时即时判断流量价值，提升定向和投放效率。[〔7〕](https://zhuanlan.zhihu.com/p/256234514)

---

## 4.2 广告主侧需要做什么

| 阶段 | 广告主侧动作 |
|---|---|
| 接入准备 | 与媒体完成RTA接口、ID匹配、加密传输、超时策略、QPS容量规划 |
| 数据准备 | 建立用户画像、交易行为、品类偏好、生命周期、会员等级、LTV、流失风险等特征 |
| 模型建设 | 训练CTCVR、GMV、LTV、复购率、流失召回等模型 |
| 实时服务 | 支持毫秒级特征查询、模型推理、策略判断 |
| 参竞策略 | 制定阈值、频控、预算、成本、ROI、商品库存等规则 |
| 系数策略 | 将CTCVR或用户价值映射为bid_factor |
| 数据回传 | 回传曝光、点击、加购、下单、支付、深度转化等事件 |
| 策略迭代 | 分媒体、分广告位、分行业目标做校准和AB实验 |

---

## 4.3 媒体侧需要做什么

| 阶段 | 媒体侧动作 |
|---|---|
| 请求触发 | 在广告请求发生时判断是否需要调用广告主RTA |
| 身份映射 | 使用合规的加密ID或授权ID与广告主进行匹配 |
| 请求发送 | 发送广告位、场景、设备、地域、计划、时间等必要字段 |
| 超时控制 | RTA广告主超时则走默认策略或放弃该广告主 |
| 竞价排序 | 结合广告主返回结果、媒体pCTR/pCVR、广告质量、出价完成排序 |
| 展示控制 | 控制频次、素材审核、体验质量、广告安全 |
| 效果归因 | 接收广告主转化回传，完成归因和模型学习 |
| 生态治理 | 防作弊、反无效流量、保护用户体验 |

# 5. 腾讯、字节等头部媒体的RTA流程和差异

以下是基于行业通用机制的抽象说明。**腾讯广告、巨量引擎/字节跳动等平台的具体接口字段、超时阈值、系数范围、支持目标、归因口径和审核要求，应以媒体最新官方对接文档和商务开通能力为准**。

---

## 5.1 共同点：头部媒体RTA大体流程一致

腾讯、字节等头部媒体的RTA通常都遵循以下通用模式：

```text
媒体广告请求
 → 调用广告主RTA服务
 → 广告主返回是否参竞/出价系数/价值标签
 → 媒体侧完成竞价排序
 → 广告展示
 → 转化回传
 → 双方模型迭代
```

共同特点包括：

1. **媒体侧控制最终竞价与展示**  
   广告主返回的是参竞意愿和价值调节信息，不代表一定展示。

2. **广告主侧更适合表达后端价值**  
   例如复购概率、会员价值、品类偏好、价格敏感度、流失风险、用户生命周期。

3. **媒体侧更适合预估平台内点击和互动**  
   例如广告位点击率、短视频内容兴趣、社交场景互动、历史广告反馈。

4. **RTA本质是联合建模和联合决策**  
   媒体负责前端流量理解，广告主负责后端转化价值判断。RTA联合出价中，广告主返回的调节系数可以修正媒体侧pCTR、pCVR或表达广告主后端收益。[〔1〕](https://zhuanlan.zhihu.com/p/385993960)

---

## 5.2 腾讯广告RTA：更强调社交生态和用户关系链场景

腾讯广告覆盖微信、QQ、腾讯视频、腾讯新闻、优量汇等多类流量场景。其RTA对广告主的价值通常体现在：

| 维度 | 特点 |
|---|---|
| 流量场景 | 社交、内容、视频、小程序、联盟等多场景 |
| 用户关系 | 社交生态强，用户身份体系和使用场景稳定 |
| 适合目标 | 召回、复购、会员转化、品牌人群再营销、私域联动 |
| 策略重点 | 用户生命周期、会员状态、历史购买、品类偏好、复购窗口 |
| 系数策略 | 对高价值存量用户、临近复购用户、会员升级用户提高系数 |

### 腾讯场景下的示例策略

假设是电商广告主做复购召回：

| 用户类型 | 广告主实时判断 | 参竞策略 | bid_factor |
|---|---|---|---:|
| 近30天浏览未购高客单商品 | 高意向、高价值 | 参竞 | 2.0 |
| 会员即将流失 | 有召回价值 | 参竞 | 1.5 |
| 刚购买同类商品 | 短期无复购需求 | 不参竞 | 0 |
| 历史低价敏感用户 | 需配合促销 | 低系数或仅活动期参竞 | 0.5 |
| 高LTV核心用户 | 长期价值高 | 强参竞 | 2.5 |

腾讯生态中，广告主可重点利用自身的 **会员、交易、私域、复购周期** 数据，媒体侧则结合社交与内容场景完成广告排序。

---

## 5.3 字节/巨量引擎RTA：更强调内容兴趣、短视频信息流和实时兴趣变化

字节系流量包括抖音、今日头条、西瓜视频、番茄小说、穿山甲等。其RTA价值通常体现在：

| 维度 | 特点 |
|---|---|
| 流量场景 | 短视频、信息流、内容推荐、联盟流量 |
| 用户兴趣 | 兴趣变化快，内容消费强驱动 |
| 适合目标 | 拉新、种草、转化、召回、直播/电商闭环联动 |
| 策略重点 | 实时兴趣、商品偏好、内容场景、价格带匹配 |
| 系数策略 | 对近期强兴趣、强购买意向、内容匹配度高的用户提高系数 |

### 字节场景下的示例策略

假设是电商广告主做商品转化：

| 用户类型 | 广告主实时判断 | 参竞策略 | bid_factor |
|---|---|---|---:|
| 站内刚浏览同款商品未购 | 强转化意向 | 强参竞 | 2.5 |
| 近期浏览同类目多次 | 中高意向 | 参竞 | 1.5 |
| 历史购买过但当前无近期行为 | 轻召回 | 低系数参竞 | 0.7 |
| 价格带明显不匹配 | 转化概率低 | 不参竞 | 0 |
| 高客单潜力用户 | CTCVR中等但GMV高 | 参竞 | 1.5 - 2.0 |

字节场景下，用户内容兴趣变化快，因此广告主RTA策略应更重视 **实时行为窗口**，例如最近1小时、6小时、24小时的浏览、搜索、加购、价格关注、品类兴趣变化。

---

## 5.4 腾讯与字节RTA策略差异总结

| 对比项 | 腾讯广告RTA | 字节/巨量引擎RTA |
|---|---|---|
| 核心流量特征 | 社交生态、稳定身份、多场景触达 | 内容推荐、短视频信息流、兴趣变化快 |
| 用户理解重点 | 用户关系、会员、社交/私域链路、复购周期 | 实时兴趣、内容偏好、短期转化意图 |
| 广告主侧重点 | 存量召回、复购、会员运营、私域联动 | 商品转化、内容种草、兴趣拉新、直播/电商转化 |
| CTCVR特征窗口 | 可重视中长期生命周期特征 | 更重视短期行为和实时兴趣 |
| 系数策略 | 高LTV、会员、复购临界用户加价 | 高兴趣、高意图、内容匹配用户加价 |
| 频控重点 | 避免社交场景过度打扰 | 避免短视频信息流重复曝光疲劳 |
| 校准重点 | 分微信/QQ/视频/联盟等场景校准 | 分抖音/头条/穿山甲/内容场景校准 |

# 6. 从人群包升级到实时预估后，策略如何变化

## 6.1 人群包模式

人群包模式大致是：

```text
广告主离线对10亿存量用户打分
 → 选出Top N人群
 → 上传媒体
 → 媒体请求时判断用户是否命中人群包
 → 命中则参竞，不命中则不参竞或低优先级
```

优点：

- 实现简单；
- 媒体调用成本低；
- 对广告主实时服务要求低。

缺点：

- 分数更新滞后；
- 无法利用实时场景；
- 无法区分同一用户在不同媒体、时段、广告位下的价值；
- 只能表达“在不在人群里”，价值粒度粗；
- 人群包上传后，广告主难以控制用户何时参与竞价、是否胜出、何时被触达。类似问题在传统人群包定向中较常见。[〔7〕](https://zhuanlan.zhihu.com/p/256234514)

---

## 6.2 实时RTA模式

实时RTA模式变为：

```text
媒体每次请求
 → 广告主实时查特征
 → 实时预估CTCVR/GMV/LTV
 → 实时判断参竞
 → 返回bid_factor
```

优势：

| 能力 | 人群包 | 实时RTA |
|---|---|---|
| 是否实时 | 否 | 是 |
| 是否能利用媒体场景 | 弱 | 强 |
| 是否能表达价值差异 | 粗粒度 | 精细化 |
| 是否能动态控成本 | 弱 | 强 |
| 是否能结合库存/价格/活动 | 弱 | 强 |
| 是否能按预算节奏调节 | 弱 | 强 |
| 是否适合高频行为变化 | 一般 | 更适合 |

# 7. 推荐的线上决策框架

## 7.1 参竞公式

可以采用如下决策：

```text
score = calibrate_media(CTCVR_adv, media, placement)

value_ecpm = score × target_CPA × 1000
```

如果是ROI目标：

```text
value_ecpm = score × predicted_AOV / target_ROAS × 1000
```

参竞判断：

```text
if 硬规则不通过:
    no_bid
elif value_ecpm < estimated_win_ecpm × safety_ratio:
    no_bid
elif pacing不允许:
    no_bid 或 降低系数
else:
    bid
```

---

## 7.2 出价系数公式

一个相对稳健的系数公式：

```text
base_factor = (score / baseline_score) ^ α
```

如果考虑用户价值：

```text
value_factor = (score × predicted_value) / baseline_expected_value
```

最终：

```text
bid_factor =
clip(
    value_factor
    × cost_control_factor
    × pacing_factor
    × media_calibration_factor
    × exploration_factor,
    min_factor,
    max_factor
)
```

其中：

| 参数 | 建议 |
|---|---|
| `α` | 0.5 - 1.0，避免分数差异被过度放大 |
| `min_factor` | 可设0.1、0.2或媒体允许下限 |
| `max_factor` | 可设2、3、5，视媒体限制和成本稳定性 |
| `cost_control_factor` | CPA超目标时下降，低于目标时上升 |
| `pacing_factor` | 消耗慢时上升，消耗快时下降 |
| `media_calibration_factor` | 腾讯、字节、联盟流量分别校准 |

# 8. 简化伪代码

```python
def rta_decision(request):
    user_id = request.user_id
    media = request.media
    placement = request.placement
    campaign = request.campaign
    product = request.product
    timestamp = request.timestamp

    # 1. 硬过滤
    if not is_valid_user(user_id):
        return {"bid": False}

    if not is_campaign_active(campaign):
        return {"bid": False}

    if is_over_frequency(user_id, campaign):
        return {"bid": False}

    if not has_budget(campaign):
        return {"bid": False}

    if not product_available(product):
        return {"bid": False}

    # 2. 实时特征
    features = get_realtime_features(
        user_id=user_id,
        media=media,
        placement=placement,
        product=product,
        timestamp=timestamp
    )

    # 3. 模型预估
    ctcvr = predict_ctcvr(features)
    aov = predict_aov(features)
    ltv = predict_ltv(features)

    # 4. 分媒体校准
    ctcvr_calibrated = calibrate_ctcvr(
        ctcvr=ctcvr,
        media=media,
        placement=placement
    )

    # 5. 计算预期价值
    if campaign.goal == "CPA":
        value_ecpm = ctcvr_calibrated * campaign.target_cpa * 1000
    elif campaign.goal == "ROAS":
        value_ecpm = ctcvr_calibrated * aov / campaign.target_roas * 1000
    else:
        value_ecpm = ctcvr_calibrated * ltv * 1000

    # 6. 预估胜出成本
    win_ecpm = estimate_win_ecpm(media, placement, campaign)

    # 7. 是否参竞
    if value_ecpm < win_ecpm * campaign.safety_ratio:
        return {"bid": False}

    # 8. 计算出价系数
    baseline_score = get_baseline_ctcvr(media, placement, campaign)
    value_factor = ctcvr_calibrated / max(baseline_score, 1e-8)

    cost_factor = get_cost_control_factor(campaign)
    pacing_factor = get_pacing_factor(campaign)
    media_factor = get_media_calibration_factor(media, placement)

    bid_factor = (
        value_factor
        * cost_factor
        * pacing_factor
        * media_factor
    )

    bid_factor = clip(
        bid_factor,
        campaign.min_bid_factor,
        campaign.max_bid_factor
    )

    return {
        "bid": True,
        "bid_factor": bid_factor
    }
```

# 9. 关键注意事项

## 9.1 CTCVR必须做分媒体校准

同一个模型分数在腾讯和字节上不一定含义一致。原因包括：

- 用户行为环境不同；
- 广告位点击率不同；
- 流量意图不同；
- 归因窗口不同；
- 转化延迟不同；
- 媒体侧优化机制不同。

因此需要：

```text
CTCVR_raw → media_calibration → CTCVR_calibrated
```

可以按：

```text
媒体 × 广告位 × 转化目标 × 商品类目 × 时段
```

建立校准表或校准模型。

---

## 9.2 不能只投最高CTCVR人群

如果只投Top高分用户，短期CPA可能很好，但会带来：

- 人群耗尽；
- 频次过高；
- 用户疲劳；
- 增量不足；
- 新用户探索不足；
- 模型越来越保守。

建议保留一定探索流量：

```text
90% exploitation：投高确定性流量
10% exploration：探索新用户、新商品、新场景
```

比例可按预算和目标动态调整。

---

## 9.3 出价系数要有上下限

如果CTCVR分数极高，但模型误差、样本偏差或异常特征导致高估，过高系数会迅速造成成本失控。

建议：

```text
bid_factor ∈ [0.2, 3.0]
```

或按媒体允许范围设置。大盘稳定后再逐步放开上限。

---

## 9.4 要建立降级策略

RTA实时链路必须考虑：

- 超时；
- 特征服务异常；
- 模型服务异常；
- ID匹配失败；
- 媒体QPS突增；
- 网络抖动。

降级策略可分为：

| 异常 | 降级方案 |
|---|---|
| 用户无法识别 | 返回默认低系数或不参竞 |
| 特征缺失 | 使用离线画像或人群包兜底 |
| 模型超时 | 使用轻量模型或规则分 |
| 服务不可用 | 全量no_bid或媒体默认策略 |
| QPS过高 | 优先保障高价值计划和核心媒体 |

# 10. 最终总结

## 10.1 参竞判断

核心不是简单判断：

```text
CTCVR高 → 参竞
CTCVR低 → 不参竞
```

而是：

```text
广告主是否愿意为这次曝光付出的最高成本
≥
这次曝光可能需要付出的市场成本
```

CPA目标下：

```text
可接受eCPM = CTCVR_adv × 目标CPA × 1000
```

ROI目标下：

```text
可接受eCPM = CTCVR_adv × 预估客单价 / 目标ROAS × 1000
```

---

## 10.2 出价系数

出价系数本质是广告主侧对媒体基础出价的价值修正：

```text
bid_factor = 用户相对价值 × 成本控制 × 预算节奏 × 媒体校准 × 探索控制
```

简单可用：

```text
bid_factor = CTCVR_adv / CTCVR_base
```

更合理可用：

```text
bid_factor =
clip(
  (CTCVR_adv × 用户价值) / 基准预期价值
  × 成本反馈系数
  × 预算节奏系数
  × 媒体校准系数,
  下限,
  上限
)
```

---

## 10.3 腾讯、字节实践差异

- **腾讯广告RTA**：更适合结合广告主会员、私域、复购周期、长期LTV做存量召回和精细化运营。
- **字节/巨量引擎RTA**：更适合结合实时兴趣、内容场景、短期行为窗口做商品转化、兴趣拉新和内容种草。
- 二者底层流程相似，差异主要在 **流量场景、用户行为特征、模型校准、频控策略、系数映射和转化目标选择**。具体接口字段和系数限制，应以媒体最新官方对接文档为准。
