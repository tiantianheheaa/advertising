- 三种模式
- RTA：广告主返回是否参竞、策略id、出价系数。  媒体预估：pctr、pcvr、sku、出价、预算控制等。
- RTB：广告主做RTA中 媒体预估的内容。
- 共建：广告主和媒体共建。广告主做召回、出价。 媒体做预估pctr、pcvr。


--- 
下文按 **“业务流程 → 核心字段含义 → 出价系数建模 → 媒体侧作用 → 风险与监控”** 展开，帮助你完整理解外投 RTA 中广告主 ID Mapping、RTA 策略 ID、出价系数的协同机制。

---

# 1. 外投 RTA 是什么？

**RTA，Real Time API / Real Time Advertising Decision**，可以理解为：  
当媒体侧产生一次广告曝光机会时，媒体会实时请求广告主或广告主的投放系统，询问：

> “这个用户、这个场景、这个广告位，你要不要投？如果要投，按什么策略投，愿意给多高的出价强度？”

广告主侧通常会在毫秒级返回：

1. **是否参与投放**
2. **RTA 策略 ID**
3. **出价系数**
4. 可选：人群标签、创意标识、落地页、实验分组、原因码等

---

# 2. 外投 RTA 的整体流程

可以抽象为以下链路：

```text
用户访问媒体 App / 网站
        ↓
媒体侧产生广告请求
        ↓
媒体识别用户 ID、广告位、上下文
        ↓
媒体向广告主 RTA 服务发起请求
        ↓
广告主进行 ID Mapping
        ↓
广告主识别用户价值、匹配投放策略
        ↓
广告主返回 RTA 策略 ID 和出价系数
        ↓
媒体侧结合自身拍卖机制计算最终竞价
        ↓
广告胜出则曝光
        ↓
曝光、点击、转化数据回传
        ↓
广告主和媒体共同优化模型与策略
```

---

# 3. 每个环节的详细含义

## 3.1 媒体侧产生广告请求

当用户打开媒体 App、进入信息流、短视频、搜索结果页或内容详情页时，媒体会产生一次广告机会。

媒体侧通常会拿到以下信息：

| 信息类型 | 示例 | 作用 |
|---|---|---|
| 用户标识 | media_user_id、oaid、idfa、imei 的加密值 | 用于识别用户 |
| 广告位信息 | app_id、slot_id、广告位尺寸、资源类型 | 判断场景价值 |
| 上下文信息 | 时间、地域、设备、网络、频道、内容类目 | 辅助判断转化概率 |
| 媒体预估信息 | pCTR、流量质量分、底价 | 用于后续拍卖 |
| 请求信息 | request_id、timestamp | 用于链路追踪和日志对齐 |

---

## 3.2 广告主 ID Mapping

**广告主 ID Mapping** 是 RTA 中非常核心的一步。

媒体侧认识的是自己的用户 ID，例如：

```text
media_user_id = M12345
```

广告主侧，例如电商、金融、游戏、品牌主，认识的是自己的用户 ID，例如：

```text
advertiser_user_id = A98765
```

两边的用户 ID 体系不同，所以需要通过某种方式把它们关联起来，这个过程就是 **ID Mapping**。

---

## 3.3 ID Mapping 的常见方式

| Mapping 方式 | 含义 | 特点 |
|---|---|---|
| 设备 ID 匹配 | 通过 OAID、IDFA、Android ID 等加密标识匹配 | 常见，但受隐私政策影响较大 |
| 手机号 / 邮箱 Hash 匹配 | 双方对手机号、邮箱做加盐或哈希后匹配 | 准确率高，但合规要求更高 |
| Union ID / Open ID | 平台生态内统一 ID | 稳定，但依赖生态闭环 |
| Cookie / Web ID | Web 场景下的浏览器标识 | 受浏览器隐私限制影响 |
| 第三方 DMP 匹配 | 通过数据管理平台做用户映射 | 依赖第三方能力与合规授权 |
| 离线 Mapping 表 | 双方周期性同步加密 ID 映射表 | 延迟较高，但线上性能好 |
| 在线实时 Mapping | 请求时实时查询 ID 关系 | 更实时，但延迟压力大 |

---

## 3.4 ID Mapping 的输出结果

广告主侧拿到媒体请求后，可能出现三类结果：

| Mapping 结果 | 含义 | 后续处理 |
|---|---|---|
| **命中已知用户** | 能映射到广告主侧用户 ID | 可以使用用户历史行为、价值、偏好建模 |
| **命中弱匹配用户** | 只能识别设备、地域、兴趣等弱特征 | 可用泛化模型或场景模型 |
| **未命中用户** | 无法识别该用户 | 可以拒投、低系数投放，或使用冷启动策略 |

例如：

```text
media_user_id = M12345
        ↓ ID Mapping
advertiser_user_id = A98765
```

广告主就可以进一步查询该用户在广告主侧的行为，比如：

- 是否访问过商品详情页
- 是否加购过
- 是否购买过
- 最近一次活跃时间
- 历史 GMV
- 所属品类偏好
- 是否高价值用户
- 是否沉睡用户
- 是否新客
- 是否会员
- 是否风险用户

---

# 4. RTA 策略 ID 是什么？

## 4.1 定义

**RTA 策略 ID** 是广告主返回给媒体的一个策略标识，用来表示：

> 这次广告请求命中了广告主侧的哪一类投放策略、哪一个人群策略、哪一个模型策略或哪一个业务目标。

它不是简单的“广告计划 ID”，也不一定等同于媒体侧的 campaign_id。  
更准确地说，它是广告主侧实时决策结果的一个编码。

---

## 4.2 RTA 策略 ID 可以代表什么？

一个 RTA 策略 ID 可能对应以下内容：

| 策略维度 | 示例 |
|---|---|
| 人群策略 | 高价值老客、新客拉新、加购未购、浏览未购、沉睡召回 |
| 商品策略 | 手机品类兴趣用户、家电高潜用户、母婴复购用户 |
| 转化目标 | 下单、支付、注册、留资、App 激活、复购 |
| 生命周期策略 | 新客获取、活跃提升、流失召回、会员续费 |
| 出价策略 | 高价值高出价、低价值低出价、探索流量策略 |
| 实验策略 | A/B Test 实验组、模型版本组、流量探索组 |
| 预算策略 | 大促预算池、日常预算池、品牌预算池 |
| 风控策略 | 排除风险流量、低可信流量降权 |

---

## 4.3 示例

假设广告主侧定义了以下策略：

| RTA 策略 ID | 策略含义 |
|---|---|
| 1001 | 高价值老客复购策略 |
| 1002 | 浏览未购买用户召回策略 |
| 1003 | 加购未支付用户转化策略 |
| 1004 | 新客拉新策略 |
| 1005 | 沉睡用户召回策略 |
| 2001 | 大促高潜人群策略 |
| 3001 | 模型探索流量策略 |

当媒体请求到来时，广告主侧判断该用户是“近 7 天浏览过手机类目但未购买”的用户，于是返回：

```json
{
  "decision": "accept",
  "rta_strategy_id": "1002",
  "bid_coefficient": 1.35
}
```

这表示：

- 广告主愿意参与这次广告机会；
- 该用户命中 **浏览未购买召回策略**；
- 广告主愿意在基础出价上提高 **35%** 的竞争强度。

---

# 5. 出价系数是什么？

## 5.1 定义

**出价系数** 是广告主返回给媒体的一个相对出价强度，用于表达广告主对当前用户、当前场景、当前广告机会的价值判断。

它通常不是一个绝对价格，而是一个乘数。

例如：

```text
最终出价 = 基础出价 × 出价系数
```

如果基础出价为 10 元 CPM：

| 出价系数 | 最终出价 | 含义 |
|---|---:|---|
| 0 | 0 元 | 不参与投放 |
| 0.5 | 5 元 | 低价值流量，降价参与 |
| 1.0 | 10 元 | 正常参与 |
| 1.5 | 15 元 | 高价值用户，提高竞争力 |
| 2.0 | 20 元 | 极高价值用户，强竞争 |

---

## 5.2 出价系数和出价的关系

常见模式有三种。

### 模式一：广告主基础出价 × RTA 出价系数

```text
bid = base_bid × bid_coefficient
```

适合广告主对用户价值有较强判断能力的场景。

---

### 模式二：媒体侧预估出价 × RTA 出价系数

```text
bid = media_estimated_bid × bid_coefficient
```

媒体侧掌握广告位质量、CTR、竞争强度，广告主侧提供用户价值修正。

---

### 模式三：广告主价值出价与媒体排序融合

```text
eCPM = bid × pCTR × quality_score
```

广告主返回的出价系数影响 bid，媒体侧再结合点击率、广告质量、用户体验等因素进行最终排序。

---

# 6. 出价系数的业务含义

出价系数本质上表达的是：

> 广告主对这次广告机会的相对支付意愿。

它通常由以下因素决定：

| 因素 | 影响方向 |
|---|---|
| 用户转化概率高 | 出价系数提高 |
| 用户客单价高 | 出价系数提高 |
| 用户生命周期价值高 | 出价系数提高 |
| 用户已接近转化 | 出价系数提高 |
| 用户近期已购买 | 可能降低或排除 |
| 用户流失风险高但可召回 | 视召回价值提高 |
| 广告位质量高 | 出价系数提高 |
| 媒体流量历史 ROI 好 | 出价系数提高 |
| 当日预算紧张 | 出价系数降低 |
| 转化成本超目标 | 出价系数降低 |
| 频控过高 | 出价系数降低 |
| 风险流量 | 出价系数降低或拒投 |

---

# 7. 出价系数如何建模和预估？

出价系数通常不是人工拍脑袋设置，而是基于模型预估和业务约束共同生成。

整体可以拆成四层：

```text
用户价值预估
   ↓
转化概率预估
   ↓
商业价值计算
   ↓
系数归一化与策略约束
```

---

## 7.1 第一步：预估用户转化概率

最常见的是预估：

| 模型目标 | 含义 |
|---|---|
| pCTR | 用户点击广告的概率 |
| pCVR | 用户点击后转化的概率 |
| pCTCVR | 用户从曝光到最终转化的概率 |
| pPay | 用户支付概率 |
| pRetention | 用户留存概率 |
| pRebuy | 用户复购概率 |
| pLTV | 用户长期价值 |

其中电商场景最常用的是：

```text
pCTCVR = pCTR × pCVR
```

也可以直接建模：

```text
pConversion = P(用户在曝光后一定时间窗口内完成购买)
```

---

## 7.2 第二步：预估转化价值

仅知道用户会不会转化还不够，还要知道转化价值。

例如：

| 价值指标 | 含义 |
|---|---|
| AOV | 预估客单价 |
| GMV | 预估成交金额 |
| 毛利 | 预估订单毛利 |
| LTV | 用户生命周期价值 |
| 复购价值 | 未来一段时间内复购贡献 |
| 品类价值 | 用户对某类商品的潜在消费价值 |

电商场景可以计算：

```text
Expected_GMV = pConversion × Expected_AOV
```

如果以利润为目标：

```text
Expected_Profit = pConversion × Expected_AOV × Gross_Margin
```

如果以长期价值为目标：

```text
Expected_LTV = pConversion × Predicted_LTV
```

---

## 7.3 第三步：计算可接受出价

不同投放目标下，可接受出价不同。

### 目标一：CPA 约束

如果广告主目标是控制 CPA：

```text
Expected_CPA = Cost / Conversion
```

为了满足目标 CPA，可以反推单次曝光可接受出价：

```text
Bid_per_impression = Target_CPA × pConversion
```

如果是 CPM 口径：

```text
CPM_Bid = Target_CPA × pConversion × 1000
```

---

### 目标二：ROI / ROAS 约束

如果广告主目标是 ROAS：

```text
ROAS = GMV / Cost
```

目标 ROAS 为 R，则最大可接受成本为：

```text
Max_Cost = Expected_GMV / Target_ROAS
```

对应曝光出价：

```text
Bid_per_impression = Expected_GMV / Target_ROAS
```

如果按 CPM：

```text
CPM_Bid = Expected_GMV / Target_ROAS × 1000
```

---

### 目标三：利润最大化

如果广告主关注利润：

```text
Expected_Profit = Expected_Revenue - Cost
```

可接受出价可以设置为：

```text
Bid = pConversion × Expected_Margin
```

其中：

```text
Expected_Margin = Expected_AOV × Gross_Margin_Rate
```

---

## 7.4 第四步：把绝对出价转换成出价系数

很多 RTA 接口返回的是系数，而不是绝对价格。

假设广告主基础 CPM 出价是：

```text
base_cpm = 10 元
```

模型预估当前用户可接受 CPM 为：

```text
model_cpm = 15 元
```

则出价系数为：

```text
bid_coefficient = model_cpm / base_cpm = 15 / 10 = 1.5
```

如果模型预估当前用户价值较低：

```text
model_cpm = 6 元
base_cpm = 10 元
bid_coefficient = 0.6
```

---

# 8. 出价系数建模的常见公式

## 8.1 基于转化概率的简单模型

适合 CPA 投放：

```text
bid_coefficient = pConversion_user / pConversion_avg
```

含义是：

- 用户转化概率高于平均水平，系数大于 1；
- 用户转化概率低于平均水平，系数小于 1。

示例：

| 用户 | pConversion | 平均 pConversion | 出价系数 |
|---|---:|---:|---:|
| A | 2.0% | 1.0% | 2.0 |
| B | 1.0% | 1.0% | 1.0 |
| C | 0.5% | 1.0% | 0.5 |

---

## 8.2 基于 GMV 的价值模型

适合电商 ROAS 投放：

```text
expected_value = pConversion × expected_AOV
```

```text
bid_coefficient = expected_value_user / expected_value_avg
```

示例：

| 用户 | pConversion | 预估客单价 | 预期 GMV |
|---|---:|---:|---:|
| A | 2% | 500 元 | 10 元 |
| B | 1% | 300 元 | 3 元 |
| C | 0.5% | 100 元 | 0.5 元 |

如果平均预期 GMV 是 2 元，则：

| 用户 | 预期 GMV | 出价系数 |
|---|---:|---:|
| A | 10 元 | 5.0 |
| B | 3 元 | 1.5 |
| C | 0.5 元 | 0.25 |

---

## 8.3 加入预算和节奏控制

实际线上不会只看用户价值，还要考虑预算消耗。

```text
bid_coefficient =
value_score
× budget_pacing_factor
× competition_factor
× risk_factor
```

其中：

| 因子 | 含义 |
|---|---|
| value_score | 用户价值分 |
| budget_pacing_factor | 预算节奏因子 |
| competition_factor | 竞争强度因子 |
| risk_factor | 风险折扣因子 |

例如：

```text
用户价值系数 = 1.8
预算节奏因子 = 0.8
竞争强度因子 = 1.1
风险因子 = 0.9
```

则：

```text
bid_coefficient = 1.8 × 0.8 × 1.1 × 0.9 = 1.4256
```

---

## 8.4 最终线上常用形式

线上通常会做上下限截断：

```text
bid_coefficient =
clip(
    value_score
    × pacing_factor
    × media_quality_factor
    × exploration_factor
    × risk_factor,
    min_coef,
    max_coef
)
```

例如：

```text
min_coef = 0.1
max_coef = 3.0
```

如果模型计算出 5.8，也会被截断为 3.0，避免过度出价。

---

# 10. 出价系数的模型类型

## 10.1 传统模型

| 模型 | 特点 |
|---|---|
| Logistic Regression | 可解释性强，适合基础 CTR/CVR 预估 |
| GBDT / XGBoost / LightGBM | 处理非线性能力强，工业界常用 |
| FM / FFM | 适合稀疏特征交叉 |
| Wide & Deep | 同时记忆规则和泛化特征 |
| DeepFM / DIN / DIEN | 适合电商行为序列建模 |
| 多任务模型 | 同时预估点击、转化、GMV、留存等目标 |

---

## 10.2 常见建模目标

不同业务目标下，模型会有所不同：

| 业务目标 | 模型目标 |
|---|---|
| 拉新 | 新客转化概率、新客 LTV |
| 促活 | 活跃概率、访问概率 |
| 复购 | 复购概率、复购 GMV |
| 召回 | 召回成功率、召回后价值 |
| 大促 | 活动转化率、预估 GMV |
| 品牌广告 | 点击率、互动率、品牌人群触达价值 |
| App 投放 | 激活率、注册率、留存率、付费率 |

---

# 11. 出价系数在媒体侧有什么作用？

## 11.1 决定是否参与竞价

如果广告主返回：

```json
{
  "decision": "reject",
  "bid_coefficient": 0
}
```

媒体侧通常认为广告主不参与本次广告机会。

如果返回：

```json
{
  "decision": "accept",
  "bid_coefficient": 1.2
}
```

媒体侧会将该广告主纳入候选广告队列。

---

## 11.2 调整广告主在拍卖中的竞争力

媒体侧可能这样计算：

```text
final_bid = base_bid × bid_coefficient
```

或者：

```text
rank_score = base_bid × bid_coefficient × pCTR × quality_score
```

因此，出价系数越高，广告主越容易赢得曝光。

---

## 11.3 辅助媒体进行流量分配

媒体侧可以通过 RTA 返回结果知道：

- 哪些用户广告主更想要；
- 哪些流量广告主愿意高价竞争；
- 哪些流量广告主不愿意买；
- 哪些策略对应高价值转化；
- 哪些广告位更适合特定广告主。

这有助于媒体侧做更精细的流量分配。

---

## 11.4 提升广告投放效率

对于广告主而言，出价系数可以让预算更多花在高价值用户上。

例如：

| 用户类型 | 固定出价 | 动态系数出价 |
|---|---:|---:|
| 高转化用户 | 10 元 | 18 元 |
| 普通用户 | 10 元 | 10 元 |
| 低转化用户 | 10 元 | 3 元 |

动态出价的优势是：

- 高价值用户提高胜率；
- 低价值用户降低浪费；
- 整体 CPA 更可控；
- ROAS 更容易提升；
- 预算分配更符合业务目标。

---

## 11.5 保护广告主真实价值判断

广告主有时不希望直接向媒体暴露绝对出价或完整用户价值，因此返回系数比直接返回价格更灵活。

例如：

```text
广告主不直接告诉媒体：“这个用户我愿意出 37.5 元 CPM”
而是返回：“该用户系数为 1.5”
```

媒体侧只知道这是一个更高价值的用户，但不一定知道广告主完整的用户价值模型。

---

# 12. RTA 策略 ID 和出价系数的关系

二者通常是配套使用的。

| 字段 | 解决的问题 | 示例 |
|---|---|---|
| RTA 策略 ID | 这个用户为什么值得投？属于什么策略？ | 加购未支付召回策略 |
| 出价系数 | 这个用户值得用多大力度投？ | 1.8 |
| 是否参与 | 要不要投？ | accept / reject |

示例：

```json
{
  "request_id": "req_20260623_001",
  "decision": "accept",
  "rta_strategy_id": "1003",
  "bid_coefficient": 1.8
}
```

解释：

- `1003`：该用户命中 **加购未支付用户转化策略**；
- `1.8`：广告主愿意在基础出价上提高 80%；
- 媒体侧会更积极地让该广告主参与竞价。

---

# 13. 一个完整案例

## 13.1 背景

某电商广告主在媒体 App 上外投，希望召回近 7 天浏览但未购买手机的用户。

广告主设置：

```text
基础 CPM 出价：10 元
目标 ROAS：5
预算：100 万元
策略：手机类目浏览未购召回
```

---

## 13.2 用户请求到来

媒体侧请求：

```json
{
  "request_id": "abc001",
  "media_user_id": "M12345",
  "slot_id": "feed_001",
  "app_id": "media_app",
  "timestamp": 1782200000,
  "device": "android",
  "city": "beijing"
}
```

---

## 13.3 广告主 ID Mapping

广告主侧映射到：

```text
advertiser_user_id = A98765
```

查询用户行为发现：

- 近 3 天浏览过手机商品；
- 加购过一款 3999 元手机；
- 未完成购买；
- 历史购买力较高；
- 所在城市物流履约能力较好；
- 当前广告位历史转化质量较好。

---

## 13.4 模型预估

模型输出：

```text
pConversion = 1.5%
Expected_AOV = 4000 元
Expected_GMV = 1.5% × 4000 = 60 元
Target_ROAS = 5
Max_Cost = 60 / 5 = 12 元
```

假设这是单次曝光机会折算后的 CPM 口径为：

```text
model_cpm = 18 元
base_cpm = 10 元
bid_coefficient = 18 / 10 = 1.8
```

---

## 13.5 广告主返回 RTA 结果

```json
{
  "request_id": "abc001",
  "decision": "accept",
  "rta_strategy_id": "1002",
  "bid_coefficient": 1.8,
  "ttl": 300
}
```

含义：

- 参与投放；
- 命中 **手机类目浏览未购召回策略**；
- 出价系数为 **1.8**；
- 该决策结果可缓存 300 秒。

---

## 13.6 媒体侧计算

媒体侧基础出价：

```text
base_cpm = 10 元
```

融合 RTA 系数：

```text
final_cpm = 10 × 1.8 = 18 元
```

如果媒体侧排序公式为：

```text
rank_score = final_cpm × pCTR × quality_score
```

那么该广告主在本次竞价中的排序分会显著提高，更容易拿到曝光。

---

# 14. 线上系统需要关注的问题

## 14.1 延迟

RTA 是实时链路，通常要求广告主在几十毫秒内返回。

如果超时，媒体侧可能会：

- 直接跳过该广告主；
- 使用默认策略；
- 使用缓存结果；
- 降低该广告主请求优先级。

---

## 14.2 Mapping 命中率

ID Mapping 命中率直接影响 RTA 效果。

关键指标：

| 指标 | 含义 |
|---|---|
| 请求量 | 媒体发来的 RTA 请求总量 |
| 可识别用户量 | 能映射到广告主 ID 的用户量 |
| Mapping 命中率 | 可识别用户量 / 请求量 |
| 策略命中率 | 命中有效投放策略的比例 |
| 参与竞价率 | 返回 accept 的比例 |

如果 Mapping 命中率低，即使模型很好，也无法充分发挥价值。

---

## 14.3 模型校准

模型输出的 pCVR、pGMV、LTV 必须校准，否则会导致出价偏差。

常见问题：

| 问题 | 后果 |
|---|---|
| 高估转化率 | 出价过高，CPA 上升 |
| 低估转化率 | 出价过低，拿不到好流量 |
| 高估客单价 | ROAS 下降 |
| 未考虑归因延迟 | 模型反馈滞后 |
| 样本选择偏差 | 模型只学到已曝光人群，泛化差 |

---

## 14.4 预算节奏

如果只按价值出价，可能导致预算过快消耗。

所以需要加入 pacing 控制：

```text
pacing_factor = 目标消耗进度 / 实际消耗进度
```

例如：

| 情况 | 处理 |
|---|---|
| 实际消耗快于计划 | 降低出价系数 |
| 实际消耗慢于计划 | 提高出价系数 |
| 接近预算上限 | 降低系数或停止投放 |
| 大促高峰期 | 放宽上限，提高竞争 |

---

## 14.5 频控与用户体验

广告主不能只看转化，还要控制用户体验。

例如：

- 用户短时间内已经看过多次广告，应降低系数；
- 用户已完成购买，应切换复购策略或排除；
- 用户对广告无互动，应降权；
- 用户反馈不感兴趣，应停止投放。

---

# 15. 关键监控指标

建议从以下维度监控 RTA 效果。

| 指标类别 | 指标 |
|---|---|
| 链路质量 | QPS、平均延迟、P95/P99 延迟、超时率、错误率 |
| Mapping | ID Mapping 命中率、可识别用户占比 |
| 策略 | 策略命中率、各策略请求量、各策略转化率 |
| 出价 | 系数分布、均值、P90、P99、截断比例 |
| 拍卖 | 参竞率、胜率、CPM、CPC |
| 效果 | CTR、CVR、CPA、GMV、ROAS、LTV |
| 预算 | 消耗进度、预算利用率、超耗风险 |
| 用户体验 | 频次、负反馈率、投诉率 |
| 模型 | AUC、LogLoss、校准度、分桶转化率 |

---

# 16. 总结

**外投 RTA 的本质是广告主实时参与媒体侧广告决策。**

核心链路可以概括为：

```text
媒体请求
→ 广告主 ID Mapping
→ 用户价值识别
→ 策略匹配
→ 返回 RTA 策略 ID
→ 返回出价系数
→ 媒体侧融合拍卖
→ 曝光和转化反馈
→ 模型持续优化
```

关键理解：

1. **广告主 ID Mapping**  
   解决“媒体用户是谁、广告主是否认识这个用户”的问题。

2. **RTA 策略 ID**  
   解决“这个用户命中了什么业务策略”的问题，例如拉新、召回、复购、大促、高价值用户等。

3. **出价系数**  
   解决“这个用户值得用多大力度竞争”的问题，通常是基础出价的乘数。

4. **出价系数建模**  
   通常基于 pCTR、pCVR、pGMV、LTV、预算节奏、媒体质量、风险控制等综合计算。

5. **媒体侧作用**  
   媒体会使用出价系数调整广告主的竞价强度、排序分和流量分配，从而影响广告是否胜出曝光。

下一步如果要落地，可以继续拆成两部分：  
**一是 RTA 接口字段设计**，包括请求参数、返回参数、超时和缓存机制；  
**二是出价系数模型方案**，包括特征、标签、样本构造、训练目标和线上校准。
