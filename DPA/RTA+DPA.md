下面按 **概念差异 → 投放展示全流程 → 广告主/媒体侧分工 → 腾讯、字节等主流媒体实践差异** 来说明。

---

## 1. 核心概念：DPA、sDPA、mDPA、RTA分别是什么

### 1.1 DPA：动态商品广告

**DPA（Dynamic Product Ads，动态商品广告）** 是一种面向多商品、多品类广告主的商品广告解决方案。它把广告主的 **商品库**、**用户行为数据** 和媒体侧的 **用户理解/推荐模型** 结合起来，在广告请求发生时，为不同用户动态选择最合适的商品、创意和落地页，实现“千人千面”的商品展示。典型场景包括电商、旅游、酒店、本地生活、教育、金融等具备结构化商品/服务库的行业。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔9〕](https://www.sohu.com/a/856505823_121384165)

DPA的“动态”主要体现在三点：**动态内容**，即不同用户看到的商品不同；**动态落地页**，即可跳转到商品详情页、品类页、聚合页或搭售页；**动态用户**，即受众会随着用户近期行为变化而持续变化，而不是一次性圈定固定人群。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

---

### 1.2 mDPA：多商品动态广告，本质上通常就是“标准DPA”

**mDPA** 通常可理解为 **Multi-product DPA / 多商品动态广告**。在很多媒体或行业语境中，mDPA并不会被单独命名，而是直接称为 **DPA、多商品广告、动态商品广告**。它的核心逻辑是：**为“人”找最合适的“商品”**，即根据用户近期浏览、搜索、收藏、加购、下单等行为，从广告主商品库中动态推荐最可能转化的商品。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)[〔4〕](https://www.woshipm.com/marketing/5113075.html)

mDPA更适合拥有 **海量SKU、商品更新频繁、用户行为数据丰富** 的广告主，例如京东、综合电商、垂类电商、酒旅平台等。它通常用于 **再营销、拉活、召回、相似商品推荐、个性化商品种草** 等场景。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)[〔4〕](https://www.woshipm.com/marketing/5113075.html)

---

### 1.3 sDPA：单商品动态广告

**sDPA / SDPA（Single Dynamic Product Ad，单商品广告）** 是DPA的衍生形态。与mDPA“为人找商品”不同，sDPA更强调 **为“优质商品”找合适的“人”**，通常用于爆品、重点单品、新品或高毛利商品的推广。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔8〕](https://www.shangyexinzhi.com/article/3872113.html)

sDPA的模型侧重点也不同：DPA通常结合广告主上传的用户行为数据和媒体侧用户理解，为用户推荐最契合的商品；SDPA则更多结合广告主历史投放数据、商品表现和媒体侧用户理解，把爆款或指定商品推送给最可能感兴趣的人群。[〔4〕](https://www.woshipm.com/marketing/5113075.html)

---

### 1.4 RTA：实时广告主决策接口，不是广告样式

**RTA（Real Time API）** 是一种实时接口机制。媒体在每次广告请求发生时，通过API把部分流量信息同步给广告主，广告主基于自身数据和策略实时判断：**是否参与、参与哪个计划、出什么价、是否排除该用户、是否推荐某些商品或人群策略**。它可以理解为让广告主更深度参与媒体广告投放决策的接口能力。[〔1〕](https://m.toutiao.com/article/7037823378530828811/?wid=1707226765320)[〔2〕](https://zhuanlan.zhihu.com/p/417326165)[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

RTA不是一种创意样式，也不是商品广告本身。它更像是媒体投放链路中的一个 **实时决策增强层**。DPA解决的是“给用户展示什么商品/创意”，RTA解决的是“这次流量广告主要不要、价值多少、是否需要用广告主自有策略干预”。两者可以独立使用，也可以组合成 **RTA+DPA**。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

---

## 2. DPA、sDPA、mDPA、RTA的核心区别

| 维度 | mDPA / DPA | sDPA / SDPA | RTA |
|---|---|---|---|
| 本质 | **多商品动态广告解决方案** | **单商品/重点商品动态推广方案** | **实时决策API机制** |
| 核心问题 | 给这个用户推荐哪个商品 | 给这个商品找到哪些用户 | 这次流量广告主要不要、价值多少 |
| 决策对象 | 商品、创意、落地页、人群 | 商品对应的潜在人群 | 是否参竞、出价、计划、商品候选、过滤策略 |
| 典型目标 | 拉活、召回、再营销、个性化推荐 | 爆品放量、新品推广、重点SKU引流 | 控制流量质量、排除无效用户、结合自有数据提升ROI |
| 主要数据 | 商品库 + 用户行为 + 媒体模型 | 重点商品 + 历史投放表现 + 媒体模型 | 广告主自有用户数据 + 媒体实时请求信息 |
| 广告主控制力 | 中等，主要通过商品库、事件、出价、规则控制 | 中等，主要控制商品选择和投放目标 | 高，可实时筛选用户和表达价值 |
| 媒体侧作用 | 商品召回、排序、动态创意、竞价、展示 | 人群扩展、商品-用户匹配、竞价、展示 | 发起请求、接收决策、纳入媒体竞价与投放链路 |
| 技术复杂度 | 中高，需要商品库、行为数据、转化回传 | 中等，需要商品信息和商品投放数据 | 高，需要低延迟API、ID匹配、实时策略系统 |
| 是否可组合 | 可与RTA组合 | 可与RTA组合 | 可与DPA/sDPA/常规广告组合 |

## 3. 全流程说明：从投放准备到广告展示

## 3.1 mDPA / 标准DPA全流程

### 阶段1：广告主准备商品库

广告主需要把商品数据同步到媒体商品库。商品信息通常包括商品ID、标题、图片、价格、类目、库存、落地页、促销信息等。同步方式通常包括 **手动录入、文件上传、定时拉取**；对于SKU多、价格和库存变化频繁的电商广告主，更适合使用定时拉取方式，以保证商品信息及时更新。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

**广告主需要做：**

- 建立结构化商品库；
- 保证商品ID稳定；
- 提供商品标题、图片、价格、类目、库存、落地页等字段；
- 维护上下架、库存、价格、促销等状态；
- 避免商品图片质量差、落地页失效、价格不一致等问题。

**媒体侧需要做：**

- 接收并校验商品库；
- 建立媒体侧商品索引；
- 做商品质量审核；
- 支持商品库更新保护，避免误删或异常更新；
- 将商品信息用于后续召回、排序和创意拼接。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

---

### 阶段2：广告主同步用户行为或转化数据

DPA需要理解用户对哪些商品、类目或服务有兴趣，因此广告主需要向媒体同步用户行为数据，例如浏览、搜索、点击、收藏、加购、下单、支付等。常见方式包括 **SDK、MAPI、RTA**。SDK方式开发成本较低，但用户意图计算更多依赖媒体；MAPI方式由广告主计算用户意图后回传，开发量较大；RTA方式则可减少直接回传用户行为明细，让广告主在实时请求时参与决策。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

**广告主需要做：**

- 明确关键行为事件：浏览、搜索、加购、收藏、下单、支付等；
- 建立用户-商品兴趣关系；
- 接入SDK、MAPI或RTA；
- 做转化回传和归因；
- 遵守隐私合规要求，避免过度传输敏感信息。

**媒体侧需要做：**

- 提供事件接入规范；
- 完成用户ID匹配；
- 建立用户兴趣、商品兴趣和转化模型；
- 支持归因、报表和优化；
- 将用户行为与商品库动态关联。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

---

### 阶段3：广告主创建DPA广告计划

商品库和行为数据准备完成后，广告主创建DPA广告计划，将 **广告目标、出价方式、预算、商品库、用户行为、创意模板、落地页规则** 关联起来。DPA相比传统广告的优势是，广告主不需要为海量SKU逐个创建广告计划，一条计划即可关联大量商品，显著降低建计划和创意制作成本。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

**广告主需要做：**

- 选择投放目标：激活、拉活、加购、下单、支付等；
- 选择商品范围：全量商品、指定类目、指定价格带、指定促销商品；
- 设置预算、出价、转化目标；
- 配置创意模板；
- 配置商品详情页、品类页或聚合页等落地页规则；
- 设置排除规则，例如已购买用户、低库存商品、不可售商品。

**媒体侧需要做：**

- 提供DPA建计划能力；
- 支持商品库与计划绑定；
- 支持创意模板自动拼接；
- 支持动态落地页；
- 支持预算控制、频控、审核、竞价和投放优化。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔9〕](https://www.sohu.com/a/856505823_121384165)

---

### 阶段4：用户产生广告请求

当用户打开媒体App、信息流、视频流、小程序或联盟流量时，媒体侧产生广告请求。媒体根据用户画像、上下文、广告位、历史行为和广告主商品库，判断是否召回该广告主的DPA计划，并从商品库中选出候选商品。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔9〕](https://www.sohu.com/a/856505823_121384165)

**广告主此时主要依赖前置配置和数据质量**，包括商品库是否准确、行为数据是否及时、出价是否合理、转化回传是否稳定。

**媒体侧需要做：**

- 识别用户和广告位上下文；
- 召回可能匹配的DPA计划；
- 从商品库中召回候选商品；
- 根据用户兴趣、商品质量、出价、预估CTR/CVR等因素排序；
- 生成动态创意并参与竞价。

---

### 阶段5：动态创意生成与广告展示

DPA广告展示时，媒体会把商品主图、标题、价格、促销信息、店铺信息等拼接成广告创意。由于商品图片尺寸、信息结构不同，媒体通常提供创意模板能力，广告主可以使用平台模板或上传模板，让系统自动生成适配不同广告位的动态创意。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

展示完成后，用户可能发生 **show、click、转化** 等行为。show表示广告曝光，click表示广告点击，CTR通常代表点击率，CVR通常代表转化率，这些指标会进入后续优化链路。[〔1〕](https://m.toutiao.com/article/7037823378530828811/?wid=1707226765320)

---

### 阶段6：点击、落地页、转化与优化

用户点击广告后，会进入商品详情页、品类页、聚合页或活动页。媒体和广告主通过归因链路记录点击、加购、下单、支付等转化行为，并持续优化商品推荐、出价、创意和人群策略。DPA的主要价值在于减少建计划成本、降低创意制作成本、提升个性化推荐能力和ROI。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

---

## 3.2 sDPA / SDPA全流程

### 阶段1：广告主选择重点商品

sDPA的起点不是“用户最近对哪些商品感兴趣”，而是“广告主要重点推广哪些商品”。广告主通常会选择爆品、新品、活动主推品、高毛利品、库存压力品或品牌战略品作为sDPA推广对象。SDPA的核心目标是让优质商品持续找到更合适的人群。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔8〕](https://www.shangyexinzhi.com/article/3872113.html)

**广告主需要做：**

- 选择重点商品或单品池；
- 提供商品信息、素材、价格、落地页；
- 明确推广目标：放量、成交、收藏、加购、品牌种草等；
- 提供历史投放数据或商品表现数据；
- 配置预算、出价和转化目标。

**媒体侧需要做：**

- 理解商品特征；
- 根据历史投放数据和媒体用户模型寻找潜在人群；
- 扩展相似兴趣人群；
- 自动生成或拼接创意；
- 在流量请求中判断哪些用户适合看到该商品。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔8〕](https://www.shangyexinzhi.com/article/3872113.html)

---

### 阶段2：媒体侧为商品找人

SDPA的模型逻辑更接近“商品驱动”。媒体侧会结合商品类目、价格、卖点、历史点击转化、用户兴趣、相似人群等特征，为该商品寻找可能感兴趣的人。相比mDPA，SDPA更适合解决“爆品如何继续放量”“预算有限如何集中打优质商品”等问题。[〔8〕](https://www.shangyexinzhi.com/article/3872113.html)

---

### 阶段3：广告展示、点击和转化优化

当媒体识别到合适用户时，会展示该重点商品的动态创意。后续点击、落地页、转化、归因和优化流程与DPA类似，但优化重心更偏向 **单品转化效率、人群扩展质量、爆品生命周期管理**，而不是在海量商品中为每个用户动态挑选不同商品。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔8〕](https://www.shangyexinzhi.com/article/3872113.html)

---

## 3.3 RTA全流程

### 阶段1：广告主建设实时决策服务

RTA要求广告主具备实时API服务能力。广告主需要基于自有数据建设用户识别、价值预估、排除策略、出价策略、频控策略、商品候选策略等能力，并保证接口低延迟、高稳定性。RTA相比普通媒体直投的关键变化是：广告主可以在投放前根据自身需求对部分用户进行排除或筛选，减少重复投放和无效投放。[〔2〕](https://zhuanlan.zhihu.com/p/417326165)[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

**广告主需要做：**

- 建设RTA服务接口；
- 建立用户ID匹配或映射能力；
- 建立实时用户价值评估模型；
- 配置是否参竞、出价、计划选择、排除规则；
- 做接口监控、超时降级、预算保护；
- 对接媒体返回日志和效果数据。

**媒体侧需要做：**

- 提供RTA协议和接入规范；
- 在广告请求发生时调用广告主RTA接口；
- 传递允许范围内的用户、广告位、上下文、计划等信息；
- 接收广告主返回的参竞、出价、过滤或商品策略；
- 将RTA结果纳入媒体自身竞价、排序、频控和展示链路。[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

---

### 阶段2：用户产生广告请求，媒体调用RTA

当用户产生广告请求时，媒体先根据自身规则判断是否需要调用广告主RTA接口。如果调用，媒体会把流量信息通过API发送给广告主，广告主实时返回决策结果，例如 **不投、参与、提高出价、降低出价、选择某个计划、返回商品候选集** 等。[〔1〕](https://m.toutiao.com/article/7037823378530828811/?wid=1707226765320)[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

RTA与RTB不同。RTB更接近广告主在AdX中直接购买流量并表达CPM出价；RTA则通常仍处在媒体原生广告投放流程下，广告主通过接口表达对流量价值的预期，其余展示、排序、频控、创意渲染等环节仍由媒体完成。[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

---

### 阶段3：媒体根据RTA结果继续竞价和展示

媒体收到广告主RTA返回结果后，会结合媒体侧预估模型、广告主出价、预算、频控、创意质量、广告位规则等继续完成竞价。如果胜出，则进行广告展示；如果未胜出或广告主返回不参与，则不展示该广告主广告。[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

---

### 阶段4：效果回传与策略迭代

展示、点击和转化发生后，媒体和广告主通过日志、归因、转化回传等方式进行效果分析。广告主会持续优化RTA策略，例如排除已购买用户、提高高LTV用户出价、降低低价值用户出价、针对流失用户做召回等。RTA更适合广告主获取新客、挽回流失用户、控制重复投放和提升ROI。[〔2〕](https://zhuanlan.zhihu.com/p/417326165)

---

## 3.4 RTA + DPA组合流程

**RTA+DPA** 是电商广告主常见的高阶玩法：DPA负责商品库、动态创意和个性化商品展示；RTA负责让广告主基于自有数据实时参与流量和商品决策。广告主可以通过RTA接口把用户偏好、商品候选、商品重要程度、个性化出价等信息反馈给媒体，媒体再结合自身模型选择更可能转化的商品进行展示。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)

公开资料中，穿山甲RTA+DPA的核心价值包括：**高ROI商品优选**，即结合媒体侧用户洞察和广告主侧商品候选集，优先选择候选集中ROI更高的商品展示；**商品重要程度**，即支持广告主对重点商品叠加权重或保障。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)

RTA+DPA的完整链路可以概括为：**用户请求 → 媒体调用RTA → 广告主实时判断用户价值和商品候选 → 媒体结合DPA模型选择商品 → 动态生成创意 → 竞价胜出 → 展示 → 点击/转化 → 数据回流优化**。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)

## 4. 每种模式下广告主和媒体侧分别做什么

| 模式 | 广告主主要职责 | 媒体侧主要职责 |
|---|---|---|
| **mDPA / DPA** | 提供商品库；同步用户行为；配置计划、预算、出价、创意模板、落地页；回传转化；维护商品状态 | 接收商品库；做人货匹配；动态生成创意；完成召回、排序、竞价、展示、归因和报表 |
| **sDPA / SDPA** | 选择重点商品；提供商品素材和落地页；配置投放目标和预算；提供历史商品表现数据 | 基于商品特征和用户模型找人；扩展相似人群；动态创意展示；优化单品转化效率 |
| **RTA** | 建实时API；做用户价值判断；返回是否参竞、出价、计划、人群或商品策略；监控接口和效果 | 在广告请求时调用RTA；传递流量信息；接收广告主决策；纳入媒体竞价和展示链路 |
| **RTA + DPA** | 同时维护商品库和RTA策略；实时返回用户价值、商品候选、商品权重、出价策略 | 同时执行RTA调用和DPA商品推荐；结合双方模型选择商品；动态创意、竞价、展示和优化 |

## 5. 国内主流媒体：腾讯、字节等DPA和RTA流程与区别

## 5.1 国内主流媒体的共性

腾讯、字节等主流媒体的DPA底层思路大体一致：广告主先接入商品库和用户行为/转化数据，媒体侧基于商品库、用户兴趣、广告位上下文、出价和转化目标进行商品推荐、动态创意生成、竞价和展示。差异更多体现在 **商品库字段规范、数据回传方式、ID体系、创意模板、审核规则、归因口径、自动化出价能力、是否支持RTA+DPA深度联动** 等产品细节上。[〔4〕](https://www.woshipm.com/marketing/5113075.html)[〔7〕](https://news.qq.com/rain/a/20210827A09GFN00)

RTA在国内主流媒体中的共性是：媒体在广告请求链路中调用广告主实时接口，让广告主根据自有数据参与流量筛选和价值判断。它通常不是替代媒体投放系统，而是在媒体原生广告链路中增加广告主决策能力。[〔6〕](https://zhuanlan.zhihu.com/p/564058606)

---

## 5.2 字节系：巨量引擎 / 穿山甲的DPA、SDPA、RTA+DPA

公开资料显示，穿山甲DPA又称多商品广告，适合拥有海量商品、希望实现目标用户与商品个性化连接的广告主，尤其适合电商行业“种草”和转化场景。其逻辑是基于目标用户洞察，为“人”推荐“商品”。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)

字节系的SDPA则更强调“让商品找人”，适合爆款单品或重点商品投放。公开资料中对DPA与SDPA的区别概括为：**DPA是为人寻找最适合的商品，SDPA是为优质商品找到懂它的人**。[〔8〕](https://www.shangyexinzhi.com/article/3872113.html)

穿山甲RTA+DPA的公开流程是：广告主基于自身数据库预估高转化商品，并通过RTA接口传递商品信息和人群偏好洞察；媒体侧借助DPA投放能力，对人群进行千人千面的商品广告展现。该组合相比普通DPA更强调高ROI商品优选和商品重要程度控制。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)

公开案例中，某电商广告主使用RTA推荐商品策略并叠加人群分层个性化出价，在拉活场景下降低付费成本；另有广告主结合自身数据和穿山甲模型共同推荐商品，CVR提升、付费成本下降。此类数据属于公开案例口径，实际效果会受行业、预算、商品质量、归因口径和投放周期影响。[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html)

---

## 5.3 腾讯系：腾讯广告DPA与RTA的典型链路

基于行业通用DPA流程，腾讯系媒体的DPA通常也可抽象为：**商品库接入 → 行为/转化数据接入 → 创建动态商品广告计划 → 媒体侧做人货匹配 → 动态创意展示 → 点击转化 → 效果回传优化**。由于不同时间、不同账户、不同流量版位的产品命名和接口能力可能变化，具体字段、白名单、接口协议和归因口径应以腾讯广告官方后台和最新技术文档为准。

腾讯系RTA也可抽象为：**媒体侧广告请求 → 调用广告主RTA接口 → 广告主实时返回参竞/不参竞、出价或策略 → 腾讯广告系统继续完成排序、竞价、展示和归因**。与DPA相比，RTA的重点不是动态生成商品创意，而是让广告主基于自有数据实时参与流量筛选和价值判断。

---

## 5.4 字节与腾讯在DPA/RTA上的主要差异：看“产品细节”，不是看“底层逻辑”

| 对比项 | 字节系 / 巨量引擎 / 穿山甲 | 腾讯系 / 腾讯广告 |
|---|---|---|
| DPA底层逻辑 | 商品库 + 用户行为 + 媒体模型 + 动态创意 | 商品库 + 用户行为 + 媒体模型 + 动态创意 |
| SDPA表达 | 公开资料中明确强调DPA与SDPA：“为人找商品” vs “为商品找人”[〔8〕](https://www.shangyexinzhi.com/article/3872113.html) | 也可存在类似单品推广/商品广告能力，但具体命名以官方后台为准 |
| RTA+DPA联动 | 公开资料明确提到穿山甲RTA+DPA，高ROI商品优选、商品重要程度控制[〔3〕](https://www.oceanengine.com/blog/618-rta-dpa.html) | 可按RTA+DPA通用方式理解，但具体支持范围以官方文档和账户权限为准 |
| 广告主核心工作 | 商品库、行为数据、RTA策略、商品权重、出价目标 | 商品库、行为数据、RTA策略、商品规则、出价目标 |
| 媒体侧核心工作 | DPA推荐、RTA调用、动态创意、竞价展示、归因报表 | DPA推荐、RTA调用、动态创意、竞价展示、归因报表 |
| 主要差异来源 | 产品接口、投放后台、流量生态、模型能力、归因口径、审核规则 | 产品接口、投放后台、流量生态、模型能力、归因口径、审核规则 |

## 6. 用一个完整广告请求串起来

### 6.1 纯DPA / mDPA链路

```text
广告主准备商品库
→ 广告主同步用户行为/转化数据
→ 广告主创建DPA计划，绑定商品库、出价、预算、创意模板
→ 用户打开媒体App，产生广告请求
→ 媒体识别用户兴趣和广告位上下文
→ 媒体从广告主商品库召回候选商品
→ 媒体预估CTR/CVR/转化价值
→ 媒体选择商品并生成动态创意
→ 参与竞价，胜出后展示
→ 用户点击，进入商品详情页或聚合页
→ 用户加购/下单/支付
→ 转化数据回流，继续优化
```

---

### 6.2 sDPA链路

```text
广告主选择重点单品/爆品
→ 广告主配置商品信息、素材、落地页、预算和出价
→ 媒体理解商品特征和历史表现
→ 用户打开媒体App，产生广告请求
→ 媒体判断该用户是否可能对该商品感兴趣
→ 媒体生成该商品的动态创意
→ 参与竞价，胜出后展示
→ 用户点击并进入该商品落地页
→ 产生转化
→ 媒体继续优化该商品的人群扩展和转化效率
```

---

### 6.3 纯RTA链路

```text
广告主建设RTA实时决策接口
→ 广告主配置用户价值、人群排除、出价和预算策略
→ 用户打开媒体App，产生广告请求
→ 媒体向广告主RTA接口发送请求
→ 广告主实时判断该用户是否值得投放
→ 广告主返回不投/投放/出价/计划等决策
→ 媒体将RTA结果纳入广告排序和竞价
→ 胜出后展示广告
→ 用户点击和转化
→ 效果日志回流，广告主优化RTA策略
```

---

### 6.4 RTA + DPA链路

```text
广告主准备商品库
→ 广告主建设RTA接口和商品推荐/用户价值模型
→ 用户打开媒体App，产生广告请求
→ 媒体调用广告主RTA接口
→ 广告主基于自有数据判断用户价值
→ 广告主返回是否参竞、出价、商品候选、商品权重
→ 媒体结合DPA模型和广告主返回结果选择最终商品
→ 媒体生成动态创意
→ 参与竞价并展示
→ 用户点击进入动态落地页
→ 产生加购/下单/支付
→ 转化数据回流，双方模型继续优化
```

---

## 7. 关键结论

1. **DPA是广告产品/解决方案，RTA是实时决策接口。**  
   DPA解决“展示什么商品”，RTA解决“这次流量要不要、价值多少、是否用广告主策略干预”。

2. **mDPA更偏“人找商品”，sDPA更偏“商品找人”。**  
   mDPA适合海量SKU和个性化推荐；sDPA适合爆品、重点商品、新品和单品放量。

3. **广告主在DPA中主要负责商品库、行为数据、转化回传和投放目标；媒体侧负责推荐、创意、竞价和展示。**

4. **广告主在RTA中控制力更强，但技术要求也更高。**  
   需要实时API、用户价值模型、ID匹配、接口稳定性和策略迭代能力。

5. **腾讯、字节等主流媒体的底层流程相似，差异主要在产品命名、接口规范、数据口径、流量生态和RTA+DPA联动深度。**  
   字节公开资料中对穿山甲DPA、SDPA、RTA+DPA的说明较明确；腾讯侧具体接入细节建议以腾讯广告官方后台和最新技术文档为准。

## 8. 建议的落地选择

| 业务目标 | 推荐模式 |
|---|---|
| 海量SKU个性化推荐 | **mDPA / DPA** |
| 加购未买、浏览未买用户召回 | **DPA + 行为数据回传** |
| 爆品、新品、重点单品放量 | **sDPA / SDPA** |
| 排除已购用户、控制重复触达 | **RTA** |
| 用广告主自有数据提升ROI | **RTA** |
| 电商大促期间做高ROI商品优选 | **RTA + DPA** |
| 既要个性化商品，又要实时控制用户价值 | **RTA + DPA** |
